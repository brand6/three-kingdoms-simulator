# Feature Landscape

**Domain:** 单角色三国历史模拟原型（政治 / 家族 / 派系）  
**Researched:** 2026-04-04  
**Overall confidence:** MEDIUM-HIGH

## 结论先行

对于这个设计空间，**原型要成立，不是先证明“内容很多”，而是先证明“玩家每旬都能感到自己作为一个具体历史人物，被关系、家族与势力政治共同塑造”**。

因此，原型阶段的 table stakes 不是全国地图、海量文本或复杂战斗，而是：

1. **稳定的单角色旬制主循环**
2. **关系 → 功绩/任命 → 派系/家族反馈的因果闭环**
3. **至少一个可感知的婚姻/家族入口**
4. **至少一个可感知的势力政治入口**
5. **至少一个可感知的历史分歧入口**

如果缺少这些，原型会退化成普通数值养成器，无法证明“个人命运嵌入势力政治”的核心价值。

---

## Table Stakes

这些是**原型必须具备**的功能。缺失其中多项，玩家很难感受到该品类的核心吸引力。

| Feature | Why Expected | Prototype Shape | Complexity | Dependencies |
|---------|--------------|-----------------|------------|--------------|
| 旬制单角色主循环 | 单角色历史模拟首先要回答“这一旬我能做什么、做完会怎样”。RTK 系 officer play 的核心也是回合式个人行动。 | 年/月/旬推进；AP/精力；每旬 2-4 次行动；旬末/月末结算 | Low-Med | 时间系统、行动系统、HUD、结算页 |
| 明确的行动菜单与即时反馈 | 玩家必须能主动选择成长、交际、政务、军事、家族行为，而不是只看事件弹窗。 | 至少 5 类行动：拜访、训练/读书、巡察/政务、接任务、休整/家族互动 | Low-Med | 主循环、角色状态、结果弹窗 |
| 角色成长与状态资源 | 品类预期里，“我是谁、我变强了什么、代价是什么”必须持续可见。 | 六维属性 + 名望/功绩 + AP/精力/压力；行动消耗与成长反馈 | Low | 数据模型、角色页、行动结算 |
| 关系系统与关键互动 | 官方竞品普遍把关系、友好/敌对、配偶/盟友作为 officer fantasy 的核心支点。没有关系层，就只剩数值跑表。 | 好感/信任/戒备；拜访、送礼、请教、结交至少 2-3 种关系动作 | Med | 角色系统、行动系统、目标选择 UI |
| 仕途/任务/任命闭环 | 单角色历史模拟的核心满足感来自“行动结果进入仕途反馈”，而不是只涨属性。 | 主公委派/家族请求任务；月末功绩评价；升迁/落选/被冷落反馈 | Med | 主循环、关系、势力数据、月末结算 |
| 派系/政治环境可见化 | 既然卖点是政治模拟，玩家必须知道谁支持自己、谁反对自己、为何失败。 | 势力页 + 派系列表；至少显示核心人物、倾向、你与其关系、对任命的影响 | Med | 势力系统、关系系统、月末评定 |
| 家族/出身基础影响 | 在这个项目里，“寒门 vs 世家”的初始处境差异不是装饰，而是体验成立条件。 | 开局出身修正；家族/士族页；举荐、议婚、任命存在门第差异 | Med | 角色身份、家族/士族数据、关系/仕途逻辑 |
| 婚姻作为政治入口 | 该设计空间里，婚姻不是浪漫支线，而是关系网络扩张与仕途开门方式。原型至少要证明这一点。 | 可查看可议婚对象；至少一次议婚判定；成功/失败影响关系、门第、支持网络 | Med | 关系系统、家族/士族系统、事件系统 |
| 简化战争接入口 | 三国题材里完全没有战争入口会显得失真；但 prototype 只需证明“战功会反哺个人命运”。 | 出征任务、战果结算、战功/名望/伤病/派系声望变化；不做完整战场 | Med | 势力系统、任务系统、角色状态、结算 |
| 历史事件与分歧入口 | 玩家会期待“我能参与历史，而不只是刷数值”。 | 4-6 个条件驱动事件；至少覆盖关系、任命、家族、势力四类 | Med | 时间推进、关系、势力、家族、事件系统 |
| 因果可解释反馈 UI | 这类游戏最怕黑箱。若玩家看不懂为什么升官/失败，原型体验会直接崩。 | 旬末/月末页明确显示变化原因：谁支持、谁反对、哪项条件不足 | Low-Med | 日志系统、结算系统、势力/关系数据 |

### Table-stakes 中最不能缺的 4 项

1. **旬制主循环**
2. **关系系统**
3. **仕途/任命反馈**
4. **派系/家族导致的可解释差异**

少了这四项，产品会更像普通三国养成面板，而不是“单角色历史模拟”。

## Differentiators

这些不是所有同类产品都必须有，但**正是本项目应该优先打出来的独特价值**。

| Feature | Value Proposition | Prototype Shape | Complexity | Dependencies |
|---------|-------------------|-----------------|------------|--------------|
| 士族/门阀作为独立层，而非普通家族皮肤 | 这是本项目区别于一般 RTK officer play 的第一特征：不是只有个人关系，而是“士族 → 家族 → 个人”三层政治。 | 5-8 个士族样本；区域影响力、友敌士族、举荐倾向、婚配开放度 | Med-High | 家族系统、关系系统、仕途系统、事件系统 |
| 单角色视角下的“全身份共底层” | 君主、文臣、武将、在野都走同一套时间/行动/关系规则，会让项目显得更统一、更可信。 | 同一行动框架下，身份只改变权限、任务来源、事件池 | Med-High | 主循环、身份系统、行动权限、任务系统 |
| 政治因果透明化 | 许多政治模拟好玩但黑箱；如果你能明确告诉玩家“谁卡了你、为什么卡”，体验会非常新鲜。 | 任命失败解释、派系支持率、门第修正、关系修正全部可见 | Med | 势力/派系系统、日志、结算页 |
| 婚姻不是收集，而是政治通道 | 同类游戏常把结婚做成关系奖励；本项目应把它做成进入网络、缓和敌意、打开仕途的手段。 | 婚配后新增姻亲链、举荐线、派系倾向变化 | Med | 家族/士族、关系、事件、仕途 |
| 关系图谱 + 正/负羁绊 | RTK8 Remake 的 Relationship Chart / Destiny 已证明“关系可视化”很适合 officer drama；本项目可做三国本地化版本。 | 关键人物图谱；亲近/仇敌/姻亲/门生故吏等关系标签可视化 | Med | 关系系统、UI、事件系统 |
| 小样本高密度 190 剧本 | 与其追求全国，不如做一块人物网络特别密的局部样本，让每个角色都能互相影响。 | 3-5 城、2-3 势力、30-50 人物、5-8 士族，事件密度高于地图规模 | Med | 数据仓库、事件系统、关系/势力网络 |
| 轻量“家族延续感”而非重继承法 | 玩家会感到自己不是一次性角色，但又不会掉进 CK 级繁杂继承模拟。 | 有子嗣/接续提示即可；重点在家名延续和婚姻价值，不做复杂法统争夺 | Med | 婚姻系统、事件系统、存档接续 |

### 最值得优先验证的 differentiators

优先顺序建议：

1. **士族/门阀三层结构**
2. **政治因果透明化**
3. **婚姻作为政治通道**
4. **全身份共底层规则**

这四项最能把项目从“像 RTK”推到“RTK + CK 三国化，但仍是单角色游戏”。

## Anti-Features

这些功能**现在不该做**。它们要么吞掉范围，要么会把产品方向拉偏。

| Anti-Feature | Why Avoid | What to Do Instead |
|--------------|-----------|-------------------|
| 全国大地图漫游与全势力宏观经营 | 会立刻把项目拖向传统 SLG / 4X，稀释单角色视角；内容量也会爆炸。 | 只做局部 190 样本，高密度关系与事件网络 |
| 完整战场 / 实时战斗 / 兵种细节大系统 | 战争实现成本极高，且容易吞掉政治、家族、关系系统的开发资源。 | 只做“出征任务 → 战果结算 → 战功/政治反馈” |
| CK 级复杂继承法、子嗣教育、家谱深模拟 | 对 prototype 来说收益远低于成本，且很容易让系统变成家谱管理器。 | 只做婚姻价值、子嗣存在、轻量接续 |
| 海量随机文本事件先行 | 文本量会掩盖系统问题，且维护成本高。 | 先做少量高价值条件事件，验证系统因果 |
| 为不同身份做完全不同玩法模式 | 会造成“每个身份都半成品”；也违背统一底层规则原则。 | 共享同一主循环，只改权限、任务源、事件池 |
| 纯数值黑箱政治 | 若玩家不知道为什么失败，会误以为系统不公平。 | 所有关键失败都给出派系/关系/门第解释 |
| 细粒度经济/商业经营 | 容易滑向资源模拟，与“个人命运嵌入政治”主命题竞争注意力。 | 内政只保留与任务、任命、战争供给有关的抽象层 |
| 全量历史人物、全量事件链 | 数据量和校验成本远超原型需要，且会拖慢迭代。 | 只做关键人物、关键士族、关键历史节点 |
| 重美术包装优先 | 这个品类原型阶段的真实风险不是不好看，而是不好玩/不可解释。 | 先做信息架构、状态反馈和系统闭环 |

## Feature Dependencies

```text
旬制时间推进
  → 行动菜单 / 行动结算
  → 角色状态变化（AP / 精力 / 压力 / 名望 / 功绩）
  → 关系变化
  → 月末任命 / 评价

关系系统
  → 拜访 / 送礼 / 请教
  → 举荐 / 任务成功率
  → 婚姻判定
  → 派系吸纳与敌对

家族 / 士族系统
  → 开局差异
  → 婚配对象池
  → 举荐与仕途修正
  → 派系倾向 / 历史事件分支

功绩 / 名望
  → 官职提升
  → 主公信任变化
  → 派系话语权变化
  → 历史事件参与资格

简化战争接口
  → 战功 / 名望 / 伤病
  → 任命与派系地位变化
  → 历史节点改写入口

事件系统
  → 把关系 / 家族 / 派系 / 战争结果串成可见剧情后果
```

### 实现顺序建议（按依赖）

1. **旬制主循环 + HUD + 行动结算**
2. **角色状态 + 成长**
3. **关系系统**
4. **任务 / 功绩 / 月末任命**
5. **势力 / 派系解释层**
6. **家族 / 士族基础差异**
7. **婚姻接口**
8. **简化战争接入**
9. **关键历史事件**

## MVP Recommendation

原型阶段建议优先证明以下 4 组能力：

1. **我能过一旬**
   - 时间推进
   - 行动选择
   - 数值/状态反馈

2. **我和谁来往，真的会改变仕途**
   - 关系系统
   - 任务与举荐
   - 月末任命/评价

3. **我的出身与政治站位，真的会改变人生路径**
   - 家族/士族修正
   - 派系支持/反对
   - 失败原因可解释

4. **我能通过婚姻、战功或事件切入历史**
   - 一条婚姻链
   - 一次出征结果
   - 数个关键事件分支

### 建议 defer 的内容

- 子嗣教育与深继承
- 完整战场玩法
- 全国地图与全量势力内容
- 大量随机文本支线
- 高度差异化身份子模式

---

## Planning Takeaways

对后续 requirements/roadmap，最重要的落点是：

- **把“政治、家族、派系”都做成会修改同一主循环结果的变量**，不要把它们做成独立展示页。
- **优先验证可解释性，不优先验证内容量。** 玩家必须知道：为什么升官、为什么被压、为什么这门婚事有价值。
- **婚姻、战功、举荐都应通向同一个结果层：仕途与政治地位变化。**
- **prototype 的成功标准不是“功能很多”，而是“15-30 分钟内玩家已明显感到自己被卷入一个具体的政治社会网络”。**
