## Locked Decisions Carried Forward

- **D-01 ~ D-04:** 新月必须强制先选 1 个主任务，整月锁定，未选任务前不得行动。
- **D-05 ~ D-08:** 任务推进统一复用现有五个基础行动，并使用 `current_value / target_value / bonus_value` 三段式进度模型。
- **D-09 ~ D-12:** 月初任务卡只显示任务名 / 发布人 / 描述 / 预计奖励；HUD 常驻任务摘要；月末先任务月报再升官弹窗；未升官时必须显示差值提示。
- **D-13 ~ D-20:** 升官是简化任命制，必须校验空缺 + 数据要求 + 任命通知；影响先收敛到任务池层级变化 + 身份/评价文本变化；主界面不常驻展示下一官职目标。

## Findings

### 1. 现有代码最适合继续采用“GameRoot 协调 + Session 持有运行时状态”

当前项目已经把主循环关键入口放进 `GameRoot.end_current_xun()`，而不是通过松散广播拼装。Phase 2.1 如果直接把月初/月末逻辑挂到 `GameRoot` 的旬推进前后，改动面最小，也最符合已建立模式。

**结论：** 不要在 Phase 2.1 先把时间系统重构成复杂事件总线；保留 `GameRoot` 作为高层协调入口，在需要时给 `TimeManager` 增补月标签/辅助方法即可。

### 2. 静态定义适合用 Godot `Resource`，并与既有 JSON 剧本样本并存

项目当前的 190 烟雾样本仍由 JSON 驱动，但 Stack/Context 已明确推荐新定义型数据使用 `Resource`。Godot 4.6 对自定义 `Resource`、导出资源属性、Autoload 访问都支持良好。

**结论：**

- 继续保留 `scenario / character / faction / city` 的现有 JSON 加载链；
- 新增 `OfficeData / TaskTemplateData / TaskPoolRuleData / PromotionRuleData / CharacterSetupPatchData` 时直接落到 `res://data/.../*.tres`；
- `DataRepository` 新增一条 Phase 2.1 Resource 读取路径，不要把这些新定义重新塞回 Phase 1 JSON 结构里。

### 3. 荀彧开局应该通过“启动补丁”改运行时入口，而不是重写原始人物定义

当前 `GameRoot.DEFAULT_PROTAGONIST_ID` 仍然写死为 `cao_cao`。设计文档要求改成荀彧，但同时又强调 Definition/Runtime 分离。

**结论：**

- 用 `CharacterSetupPatchData` 覆盖默认主角、初始官职、初始 merit/fame/trust；
- `GameRoot.bootstrap_default_entry()` 读取 patch 后决定本局默认主角与起始 career/task 状态；
- 不要在 `characters.json` 里直接把“系统默认主角”硬编码死，避免未来 scenario patch 扩展困难。

### 4. 月任务状态应进入 `GameSession`，不是 HUD 本地变量

Phase 2 的 HUD 现在只读 `GameRoot.current_session`。Phase 2.1 的任务锁定、进度、月报结果都需要跨三旬持续存在，因此必须进运行时状态。

**结论：** 新增以下运行时对象并挂在 `GameSession`：

- `PlayerCareerState`
- `MonthlyTaskState`
- `TaskProgressSnapshot`
- `MonthlyEvaluationResult`

HUD 只读这些状态，不自己保存“当前主任务”真值。

### 5. 任务推进应挂在现有 action resolver 结果之后，而不是新增任务专用行动

这是 D-05 的硬约束。现有 `GameRoot.execute_phase2_action(...)` 已返回结构化结果，最稳定的接点是：行动结算成功后，立刻把 `action_id` 和结果交给 `TaskSystem` 更新任务进度。

**结论：**

- 继续复用 `train / study / inspect / visit / rest`；
- `TaskSystem` 根据任务模板上的 `progress_rule_id` 与动作 ID 累加统一 `current_value`；
- 不要创建“任务专属按钮”或“任务内行动菜单”。

### 6. 月末结算需要拆成两个系统：任务结算与升官判定

设计稿和 CONTEXT 都把月报与升官弹窗拆开。最自然的职责分割是：

- `TaskSystem`：结算 success / excellent / failed，产出 merit/fame/trust delta 与任务月报文本；
- `CareerSystem`：读取当前官职、promotion rule、空缺状态、任命通知来源，决定是否升官并返回升官结果；
- `GameRoot`：在第三旬结束时按固定顺序调用二者并把结果写回 session。

### 7. UI 不需要新场景切换，继续用 `MainScene` 内弹窗即可

UI-SPEC 和 Phase 2 现有模式一致：

- 月初任务选择：`ConfirmationDialog`/`PopupPanel` 弹在 HUD 上层；
- 月末任务月报：独立面板或 `AcceptDialog`；
- 升官反馈：第二个弹窗单独显示；
- 右侧任务区改为常驻摘要：当前主任务 + 当前进度 + 剩余旬数。

**结论：** 不要新建“月份管理场景”或跳页流程。

### 8. 自动化验证应延续 Phase 2 的 headless SceneTree 回归方式

现有 `phase2_xun_loop_regression.gd` 等测试已证明：使用 `extends SceneTree` + `MAIN_SCENE.instantiate()` 的 Godot headless 回归最适合此项目。

**结论：** Phase 2.1 至少需要三类自动化验证：

1. **Bootstrap/data regression**：默认主角为荀彧、Phase 2.1 资源可查询、月初候选任务可生成；
2. **Monthly career regression**：锁定任务、推进三旬、月末结算、奖励写入、达标时升官；
3. **Monthly HUD regression**：月初强制选任务、未选任务前禁止行动、HUD 摘要刷新、任务月报与升官弹窗顺序正确。

## Don’t Hand-Roll / Don’t Do

- 不要新增任务专属行动（违背 D-05）。
- 不要把月任务状态塞进 `MainHUD.gd` 局部变量。
- 不要把升官结果和任务月报压成同一个单屏弹窗（违背 D-11）。
- 不要在本 phase 提前实现推荐/反对链、多人竞争、派系阻挠、复杂政治解释。
- 不要把“下一官职条件摘要”常驻塞回主 HUD（违背 D-20）。
- 不要通过修改 `CharacterDefinition` 原始 JSON 来模拟月内成长；所有变化必须进运行时状态。

## Common Pitfalls To Plan Around

1. **默认主角只改 UI 不改 bootstrap**  
   会导致回归仍以曹操进入。

2. **第三旬结束后先 advance 再结算**  
   会把月报归到错误月份。

3. **只做任务成功，不做失败差值提示**  
   会直接违反 D-12 / UI-03。

4. **HUD 先允许行动后补选任务**  
   会直接违反 D-03。

5. **把任务池变化做成即时复杂权限变更**  
   会越界到 Phase 3。

## Validation Architecture

### Minimum automated checks

1. `phase21_bootstrap_month_gate_regression.gd`
   - boot protagonist is `xun_yu`
   - current office is `office_congshi`
   - month start produces 2–3 candidate tasks
   - action button stays blocked before task selection

2. `phase21_monthly_career_regression.gd`
   - selected task stores `current_value / target_value / bonus_value`
   - real actions update progress
   - third xun triggers month-end evaluation
   - merit/fame/trust deltas write back to runtime state
   - promotion success/failure reason payload exists

3. `phase21_monthly_hud_regression.gd`
   - month-start picker auto-opens
   - task summary rail shows task name + progress + remaining xun
   - month report opens before promotion popup
   - failed promotion shows missing-value hint line

## Planning Recommendation

Split execution into **6 plans / 4 waves**:

- **Wave 1:** contracts
- **Wave 2:** resource data and bootstrap/data loading (parallel)
- **Wave 3:** monthly task/career backend
- **Wave 4:** HUD month-start/month-end UI and regressions

This keeps each executor run below the context cliff and respects the project’s existing “typed runtime state + headless Godot regression” pattern.
