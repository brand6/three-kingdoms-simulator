# Phase 02.1: 官职与任务部署 - Context

**Gathered:** 2026-04-07
**Status:** Ready for planning

<domain>
## Phase Boundary

本阶段在现有单角色主 HUD、旬制行动与关系闭环之上，补上一个最小可玩的仕途月度闭环：默认主角切换为荀彧；新月开始时领取 1 个主任务；玩家在当月三旬内用现有基础行动推进任务；月末统一结算任务结果、功绩/名望/信任变化；满足条件时触发简化任命式升官；下月任务池据此刷新。它负责验证“事务 → 评价 → 任命”的基础仕途体验，不负责 Phase 3 的推荐/反对链、派系阻挠、多候选人竞争或复杂政治解释。

</domain>

<decisions>
## Implementation Decisions

### 月度节奏与流程门控
- **D-01:** 新月份开始时，任务选择界面必须**自动弹出且必须先选定 1 个主任务**，之后玩家才能开始本月行动。
- **D-02:** 月末结算在**当月第 3 旬结束后自动触发**，先完成任务结算与任命判定，再进入下月第 1 旬。
- **D-03:** 若玩家尚未选定本月主任务，则**不得进行本月旬内行动**；Phase 2.1 不允许“先行动后补领任务”或“追溯补记进度”。
- **D-04:** 当月主任务一经选定即**整月锁定，不允许更换、撤销或重选**。

### 任务推进与结算模型
- **D-05:** Phase 2.1 **不新增任务专属行动**；四类任务全部复用现有五个基础行动（拜访、训练、读书、巡察、休整）来推进。
- **D-06:** 所有任务统一采用**`current_value / target_value / bonus_value`** 三段式进度模型；月内推进时只累计统一进度值，不为每个任务类型单独做一套结构。
- **D-07:** `优秀完成` 的唯一判定方式是**达到 `bonus_value` 阈值**；不追加人工评语加权或额外事件链判定。
- **D-08:** 当月结束时，若主任务**未达到成功阈值，则直接记为失败**；Phase 2.1 不区分“未完成”与“失败”的复杂层级。

### 任务与月报界面
- **D-09:** 月初任务选择界面中，每张任务卡最少显示：**任务名称、发布人、任务描述、预计奖励**；难度、推荐方向与详细阈值不作为首批必显信息。
- **D-10:** 月内 HUD 的任务摘要区固定显示：**当前主任务、当前进度、剩余旬数**，用于支撑玩家每旬决策。
- **D-11:** 月末反馈采用**两段式结构**：先显示任务结果月报，再单独弹出升官结果弹窗；不把全部反馈压缩在同一张单屏报告里。
- **D-12:** 若本月未升官，月报中最重要的未达标反馈是**距离目标官职还差多少关键数值**；对 planner/researcher 来说，这意味着月报必须支持展示差值提示而不依赖复杂政治说明。

### 简化任命式升官
- **D-13:** Phase 2.1 的升官采用**简化任命制**，不是固定路线下“功绩够了自动升级”。
- **D-14:** 一次任命至少同时校验三项：**目标官职当前有空缺；角色达到该官职的数据要求；月末收到任命通知**。
- **D-15:** Phase 2.1 只做**单候选人、单上级通知**的最小任命版本；不做多人竞争、推荐/反对链、派系阻挠、复杂落选解释。
- **D-16:** 官职要求采用**数据表配置**，不同官职可分别要求功绩、声望，或两者组合；实现上不得把“只看功绩”写死在逻辑中。
- **D-17:** 任命通知的发送人采用**按官职归属配置**的方式，可由君主、太守或直属上级分别发出，而不是全局固定一个来源。
- **D-18:** 升官在 Phase 2.1 的核心影响先收敛为：**任务池层级变化 + 身份/称呼/评价文本变化**；不提前扩展到 Phase 3 风格的权限政治层。
- **D-19:** 升官后若任务池发生变化，**不需要在当月反馈中立刻详细解释**，留到下月任务选择阶段自然体现差异。
- **D-20:** 主界面平时**只显示当前官职**，不常驻展示“候补官职条件摘要”或“下一官职目标”。

### the agent's Discretion
- 各任务模板把现有五个基础行动折算为进度值的具体权重与公式，只要保持“统一进度值阈值模型”即可。
- 月初任务选择弹窗、月报面板与升官弹窗的具体节点结构、尺寸和视觉层级。
- “差多少关键数值”在未升官反馈里的具体文案样式（如缺少功绩、声望或任命通知条件时的句式）。
- 官职空缺的最小实现方式（固定配置空缺槽、简单占位状态或单角色样本化空缺表），只要不引入多人竞争逻辑。

</decisions>

## Existing Code Insights

### Reusable Assets
- `three-kingdoms-simulator/scripts/ui/MainHUD.gd` — 已有主 HUD、右侧任务区、行动相关弹窗、旬末总结弹窗；Phase 2.1 可在此挂接月初任务选择、月报与升官反馈。
- `three-kingdoms-simulator/scripts/autoload/GameRoot.gd` — 当前负责默认开局、行动执行与结束本旬；适合作为月初任务生成、月末结算与任命判定的高层协调入口。
- `three-kingdoms-simulator/scripts/runtime/GameSession.gd` — 已保存 `scenario_id`、时间、主角 ID、运行时角色状态与旬内行动历史，可扩展月度任务/仕途状态而无需另起全局入口。
- `three-kingdoms-simulator/scripts/runtime/RuntimeCharacterState.gd` — 已持有 `merit`、`fame` 等数值，可继续补入信任/仕途相关字段或挂接独立 career/task state。
- `three-kingdoms-simulator/scripts/systems/Phase2ActionCatalog.gd` 与 `three-kingdoms-simulator/scripts/systems/Phase2ActionResolver.gd` — 现有五个基础行动与结算输出可直接作为任务推进的数据来源。
- `three-kingdoms-simulator/scripts/autoload/DataRepository.gd` — 已负责场景/人物/势力/城市查询与 bootstrap_session，可作为荀彧开局 patch、官职定义和任务定义的读取入口。

### Established Patterns
- 当前项目已形成 **Autoload 管理器 + RefCounted Runtime State + MainHUD 统一渲染** 的模式，Phase 2.1 应继续沿用，不应切到场景分裂式实现。
- 现有系统强调 **Definition/Runtime 分离**；新增官职、任务、升官规则应使用静态定义对象，月任务状态与月末结果应放到运行时状态中。
- 当前时间推进以**旬**为底层单位；Phase 2.1 应在既有 xun-end 流程外层补上 month-start / month-end 钩子，而不是重写时间系统。

### Integration Points
- `GameRoot.bootstrap_default_entry()` 当前仍写死 `DEFAULT_PROTAGONIST_ID := "cao_cao"`，Phase 2.1 需要在这里接入荀彧默认开局与补丁应用。
- `GameRoot.end_current_xun()` 是接入“第 3 旬结束后自动触发月末结算”的直接位置。
- `MainHUD` 右侧任务区当前是占位文本，最适合作为“当前主任务 + 进度 + 剩余旬数”的常驻摘要位。
- 现有 `CharacterSelectorDialog` / `CharacterProfilePanel` 已可继续服务与任务相关的人物选择或查看，但本 phase 不应为任务再建一套重复选择器。

<specifics>
## Specific Ideas

- 用户确认：Phase 2.1 的核心体验是**荀彧开局的月度仕途责任制**，而不是继续只做旬内自由行动。
- 用户明确要求：**新月必须先领 1 个主任务再开始行动**，这让任务成为当月正式事务，而不是可选附加目标。
- 用户明确要求：**任务全部复用现有五个基础行动推进**，不新增任务专属行动入口。
- 用户明确要求：**升官改为简化任命制**，必须包含空缺、达标与任命通知三个要素，但保持单候选人、无政治阻挠的最小范围。
- 用户明确要求：主界面**不常驻显示下一官职目标或候补条件**，只保留当前官职常驻显示。

</specifics>
