## Summary

03.1 不是新功能 phase，而是一次“把已有 Phase 1 实现重新变成可审计事实”的收敛工作。当前审计失败点非常具体：Phase 1 的 9 个 requirement 只有 summary claim，没有正式 verification；同时 `MainHUD` 已从 Phase 1 的 value-only 时间契约漂移到自行基于 `session.current_year/current_month/current_xun` 拼字符串，导致 `CORE-02` 的共享时间真源失效。代码审查还显示 `phase1_topbar_time_regression.gd` 已明显过时：它仍断言旧文案（`当前城市`/`当前身份`/`OfficeLabel`），与当前 `MainScene.tscn` / `MainHUD.gd` 结构不再一致。

这意味着本 phase 的最优实现不是“大修 HUD”，而是做三件最小但强约束的事：第一，恢复 `TimeManager` 为唯一时间格式真源，HUD 只消费其结果；第二，建立“双层 gate”——保留真实冷启动 headless regression，同时新增一个聚焦的静态契约检查，证明 HUD 没再复制第二套格式逻辑；第三，在 `01-VERIFICATION.md` 中把九项 requirement 逐条落成 requirement → evidence → runnable command 的正式闭环。现有仓库已经有足够多的可复用资产（`TimeManager.gd`、`GameRoot.gd`、`DataRepository.gd`、Phase 1 summaries、现有回归壳），所以规划应偏向“重用并对齐”，而不是新增复杂基础设施。

**Primary recommendation:** 规划成一个小型 gap-closure wave：先重基线化 Phase 1 冷启动回归与静态时间契约 gate，再产出 `01-VERIFICATION.md`，最后仅补会阻塞九项 requirement 证据闭环的最小代码修复。

### Pattern 1: Single formatter source for visible time
**What:** 所有可见时间文本都从 `TimeManager` 的 formatter 得出，`MainHUD` 只做消费，不再根据 `session.current_year/current_month/current_xun` 自行拼接。
**When to use:** 任何显示当前年月旬、跨旬回归断言、冷启动 placeholder/success state 对齐时。
**Example:**
```gdscript
# Source: repo contract from TimeManager.gd + https://docs.godotengine.org/en/4.6/tutorials/scripting/singletons_autoload.html
func _time_manager() -> Node:
	return get_node("/root/TimeManager")

func _time_text(_session: GameSession) -> String:
	return str(_time_manager().call("get_current_label"))
```

### Pattern 2: Dual-gate verification
**What:** 一层真实冷启动 headless regression 证明运行结果；一层静态/契约 gate 证明 `MainHUD` 的实现方式没有再次分叉。
**When to use:** 任何 touching `MainHUD.gd`, `TimeManager.gd`, `MainScene.tscn`, or Phase 1 verification docs 的计划。
**Example:**
```gdscript
# Source: repo-specific recommendation derived from audit + MainHUD drift
var source := FileAccess.get_file_as_string("res://scripts/ui/MainHUD.gd")
if source.contains("时间：%d年 %d月 第%d旬"):
	push_error("MainHUD still hand-rolls the time format instead of delegating to TimeManager.")
	quit(1)
if not source.contains("get_current_label"):
	push_error("MainHUD no longer consumes TimeManager.get_current_label().")
	quit(1)
```

### Pattern 3: Requirement → evidence → command verification rows
**What:** `01-VERIFICATION.md` 每条 requirement 都要有“实现证据 + 运行命令/检查方法 + 结论”，不能只引用 summary frontmatter。
**When to use:** 全九项 requirement closure。
**Example:**
```markdown
| Requirement | Evidence | Command | Result |

```

### Anti-Patterns to Avoid
- **在 `MainHUD` 里继续手写年月旬格式：** 这会直接重新制造 `CORE-02` drift。
- **只补文档不补可执行证据：** audit 已证明 summary-only claim 不够。
- **保留 stale regression 断言：** 当前 `phase1_topbar_time_regression.gd` 仍断言旧 `OfficeLabel` / 旧 top-bar 文案，必须先重基线化。
- **顺手做 unrelated debt cleanup：** default protagonist duplicated 之类只有在阻塞九项证据闭环时才处理。

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| 时间格式统一 | HUD 内嵌第二套 `%d年 %d月 第%d旬` formatter | `TimeManager.get_current_label()` / `get_xun_label()` | 单一真源才可回归、可审计。 |
| 冷启动验证 | 只看静态文档或 mock UI | 现有 `phase1_topbar_time_regression.gd` 的真实 MainScene headless 启动壳 | 需要证明真实 boot → session → HUD 结果。 |
| 契约证明 | 复杂 AST/编辑器插件级静态分析 | 一个最小 repo-local source scan / use-site check | 本 phase 只需稳定证明“未复制 formatter”，不需要建设重型 lint 基础设施。 |
| Requirement traceability | Summary/frontmatter 拼凑结论 | 正式 `01-VERIFICATION.md` 表格 | 这是审计要求，不是风格偏好。 |

**Key insight:** 这个领域里最危险的“手搓”不是新系统，而是偷偷在 HUD、回归脚本、总结文档里各自维护一份相似但不一致的契约。

## Common Pitfalls

### Pitfall 1: 只修显示结果，不修共享来源
**What goes wrong:** runtime regression 可能重新变绿，但 HUD 依旧直接读 `session.current_year/current_month/current_xun` 拼字串。
**Why it happens:** 只关注“屏幕上看起来对”，没验证 formatter 来源。
**How to avoid:** 静态 gate 必须同时做 positive check（调用 `get_current_label`）和 negative check（禁止旧格式字符串）。
**Warning signs:** `MainHUD.gd` 仍存在 `%d年 %d月 第%d旬`、`session.current_year` 等直接格式拼接路径。

### Pitfall 2: 沿用 stale Phase 1 regression expectations
**What goes wrong:** 回归脚本失败的原因变成“断言过时”，而不是功能真实回退。
**Why it happens:** `phase1_topbar_time_regression.gd` 还在断言旧 top-bar 文案与 `OfficeLabel` 节点；当前场景结构已经变化。
**How to avoid:** 先按今天的真实 HUD contract 重建 expected labels，再把 contract 锁住。
**Warning signs:** 回归仍断言 `当前城市` / `当前身份` / `当前官职` 或访问不存在的 `OfficeLabel`。

### Pitfall 3: 把 summary 当 verification
**What goes wrong:** Requirements 在 `REQUIREMENTS.md`/summary 里“看起来完成”，但 milestone audit 仍判 orphaned。
**Why it happens:** 缺少 requirement → evidence → command 的正式表格。
**How to avoid:** `01-VERIFICATION.md` 每条 requirement 都写明代码证据、命令、结果、是否人工检查。
**Warning signs:** 证据栏只写“见某某 summary”而没有实际代码/命令。

### Pitfall 4: 借 gap-closure phase 扩 scope
**What goes wrong:** 03.1 被拖成 Phase 1/2/3 通用重构，计划失焦。
**Why it happens:** 看到旧债就想顺手清。
**How to avoid:** 只收录直接影响九项 requirement、冷启动稳定性、时间契约证据的修复。
**Warning signs:** 任务开始涉及 UI 重排、scene 重构、default protagonist 大范围清理、旧 popup 全量删除等与 audit blocker 无直接关系的工作。

## Open Questions

1. **静态契约 gate 的最小可信实现到底多严？**
   - What we know: 仅靠 runtime 结果不足以证明 HUD 没复制 formatter；D-03 明确要求双层 gate。
   - What's unclear: 用 forbidden-string scan 就够，还是要同时断言 `_time_text()` / use-site 明确调用 `TimeManager.get_current_label()`。
   - Recommendation: 同时做一条 positive delegation check + 一条 negative forbidden-format check，避免单边检查被轻易绕过。

2. **`01-VERIFICATION.md` 应以“链接旧 summary”为主，还是“复制旧 summary 内容”为主？**
   - What we know: 旧 summaries 已有大量实现事实，但缺正式 verification 结构。
   - What's unclear: 复制太多会让 verification 变成第二份 summary；链接太少又可能显得证据薄弱。
   - Recommendation: 以“短证据行 + 指向 summary/code/command”的方式组织，不复制大段 narrative。

3. **本地 4.6.1 mono 环境与官方 4.6.2 目标基线的 patch 差异是否需要显式处理？**
   - What we know: 官方当前稳定版是 4.6.2，但本机查询到 4.6.1.stable.mono。
   - What's unclear: 03.1 的 regression 是否会遇到 patch-level 差异。
   - Recommendation: 计划中把 4.6.2 记为目标基线；若命令行为/回归结果出现引擎相关歧义，再补一次 4.6.2 复跑，不要提前把 phase 扩成引擎升级任务。
