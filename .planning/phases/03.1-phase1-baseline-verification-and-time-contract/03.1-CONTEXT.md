# Phase 03.1: Phase 1 基线验证与时间契约修复 - Context

**Gathered:** 2026-04-09
**Status:** Ready for planning

<domain>
## Phase Boundary

本阶段是一个 gap-closure / verification 收敛 phase：它为已交付的 Phase 1 入口、数据骨架与角色基线补齐正式 verification 证据，并把 `TimeManager -> MainHUD` 的时间显示重新收敛为可回归、可审计的单一契约来源。它负责关闭 audit 中的 9 个 Phase 1 orphaned requirements、`CORE-02` 的时间契约漂移，以及冷启动 HUD 基线验证缺口；不负责扩展新玩法、重做 HUD 信息架构，或顺手开展与 milestone closure 无关的 Phase 1 重构。

</domain>

<decisions>
## Implementation Decisions

### 时间显示契约
- **D-01:** `TimeManager` 是时间显示的**唯一格式真源**。`MainHUD` 不再自行拼接年月旬展示格式；若 HUD 需要前缀或包装，也必须建立在同一真源契约上，而不是复制第二套年月旬格式逻辑。

### Phase 1 复验范围
- **D-02:** Phase 03.1 不是“只补一份文档”，而是对 **9 个 Phase 1 requirement 全量重核**：`CORE-01`、`CORE-02`、`DATA-01..04`、`CHAR-01`、`CHAR-02`、`CHAR-04` 都要建立 requirement → evidence → verification 的正式闭环。

### 回归测试口径
- **D-03:** 时间契约与冷启动 HUD 基线采用**双层 gate**：一层保留 headless cold-start / runtime regression，证明真实启动结果正确；另一层增加静态/契约检查，证明 `MainHUD` 依赖 `TimeManager` 的唯一时间真源，而不是继续内联第二套格式逻辑。

### 技术债处理边界
- **D-04:** 对 audit 提到的 Phase 1 技术债（如 default protagonist duplicated），只在**它影响 Phase 1 九项复验、冷启动稳定性或时间契约证据**时才纳入 03.1 修复；若不影响本次 milestone closure，则仅记录为后续 debt，不在本 phase 扩 scope。

### the agent's Discretion
- `01-VERIFICATION.md` 的具体章节结构、证据表字段顺序与 requirement-to-evidence 命名方式。
- 双层 gate 中静态契约检查采用何种最小实现（字符串断言、脚本扫描、或更明确的 helper/use-site 检查），只要能稳定证明 HUD 未再复制时间格式逻辑。
- Phase 1 九项 requirement 的证据组织顺序，以及是否将部分既有 summary / regression 重新提炼为 verification 表中的直接证据行。

</decisions>
