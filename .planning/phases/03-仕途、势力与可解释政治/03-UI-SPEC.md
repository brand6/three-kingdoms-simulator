## Phase 3 Visual and Interaction Contract Notes

### Pre-populated decisions

- Source: `03-CONTEXT.md` D-01 to D-14.
- Month-start task cards must use a medium-density first screen: task name, source type, issuer, linked character/faction, expected reward, political risk.
- Political reward/risk must appear as scan-friendly label phrases, not raw scores or long prose.
- Main HUD politics stays capped at 3 persistent items: major recommender, major blocker, current opportunity or qualification shortfall.
- Full faction attitude, office opportunity, faction structure, and player position move into detail panels rather than crowding the HUD.
- Appointment feedback uses a five-layer cause tree: qualification, vacancy/system, recommendation, resistance, competition.
- Player-facing month-end UI shows one top-line result, 2–3 matched reason lines, and 1 next-month advice line.
- If several failure causes apply, the headline uses the earliest blocking layer.
- Office changes must visibly change available task sources and unlock 0–2 representative office-only actions.
- From Phase 3 onward, actions with no office permission are hidden; actions with permission but temporary blockers stay visible and disabled with a reason.

### Layout contract

1. Keep the existing single-scene overlay architecture from `MainScene` and `MainHUD`; do not switch scenes for faction, politics, task-picking, or month-end explanation flows.
2. Preserve the current three-column rhythm:
   - left = player state,
   - center = summaries,
   - right = current task and recent events.
3. Replace the center column’s placeholder faction summary with a politics-first stack in this order:
   - 主要推荐人
   - 主要阻力
   - 当前机会 / 资格短板
4. Primary focal point for the main HUD is the center politics-first summary stack; secondary focal point is the currently selected month-task card in the right column.
5. Keep each center summary card to one heading plus 2–4 short lines; if a panel needs more than that, route to popup/detail UI.
6. Enable the existing `FactionButton` in the bottom bar for Phase 3 and treat it as the entry to a faction overview popup, not a scene transition.

### Task source card contract

Each month-task candidate card must render in this order:

1. task name
2. source type
3. issuer/requester
4. linked character or faction
5. one-line objective
6. expected reward summary
7. 2–4 political tags split between support/opportunity and risk/opposition

Card states:

- default: dominant surface + neutral border
- hover/focus: slightly darker surface only
- selected: accent border + accent-tinted fill
- unavailable: not used here; month candidates shown to player must all be selectable

### Month-end explainable-politics contract

Month report and appointment popup remain sequential: month report first, appointment result second.

Month report must add a dedicated politics section in this order:

1. top-line conclusion: 任命 / 升迁 / 驳回 / 错失机会
2. reason lines: exactly 2 or 3 hit reasons from the five-layer tree
3. political forces line: who recommended, who resisted, which bloc tilted neutral/supportive/opposed
4. next-month advice line: one explicit action-oriented sentence

Reason line style rules:

- one sentence each
- actor first when available (`荀攸举荐你…`, `旧吏群体对你仍存疑…`)
- no backend score dumps
- blocker terms must match requirement language: merit, trust, support, opposition, vacancy, qualification, competition

### Faction overview contract

The first page of the faction popup must answer “我在局中的位置是什么”. Show sections in this order:

1. player position summary
2. faction blocs / internal groups
3. ruler and core officers
4. controlled cities and strategic posture
5. faction resources relevant to politics and simplified warfare feedback

Interaction rules:

- every named major officer row must be inspectable into `CharacterProfilePanel` or its Phase 3 extension
- bloc rows must expose stance toward player as one of: 支持 / 观望 / 反对
- do not show a grand-strategy map or ruler-only personnel console

### Disabled and hidden state contract

- Hidden state = player lacks office permission entirely.
- Disabled state = player has permission but is temporarily blocked this xun.
- Every disabled action row must include a short reason after an em dash, e.g. `监察属吏 —— 本旬无可监察目标`.
- Never use disabled rows to communicate permanent role restrictions in Phase 3.

### Interaction density and click budget

- Reach key political information within 3 clicks from HUD.
- Reach faction overview in 1 click from the bottom bar.
- Reach a recommender/blocker profile in 2 clicks max from HUD summary or faction popup.
- Reach month-task confirmation in 2 clicks max: open picker → select card → confirm.

### Motion and feedback

- No animation-heavy transitions; reuse current popup-centered behavior.
- Visual emphasis comes from layout order, accent border, and copy hierarchy, not motion.
- After month-task selection, immediately refresh the right task panel and center political summary in the same frame/deferred layout pass.

### Accessibility and readability rules

- Minimum button/action row height: 48px.
- Summary cards and dialogs must wrap long Chinese copy; avoid truncating cause lines.
- Political tags must rely on text plus color, not color alone.
- Keep body copy to 2 lines in HUD cards when possible; overflow belongs in detail panels.
