## Implementation Decisions

### FACT-01 正式清除标准
- **D-01:** `FACT-01` 只有在**重跑后的 `03-VERIFICATION.md` 正式翻绿**后才算关闭；`03-11-SUMMARY.md` 仅是实现证据，不是最终 gate。
- **D-02:** `FACT-01` 的复验证据必须写成**完整证据链**：`FactionDefinition` typed field → generated `factions.json` posture data → `FactionSystem` / `GameRoot` overview payload → `FactionPanel` popup render → `phase3_politics_hud_regression.gd` 通过。
- **D-03:** 03.2 必须**显式说明历史漂移**：此前若干 Phase 3 summary 曾 claim `FACT-01` 完成，但正式 verification 一直未翻绿；只有 **03-11 实现补齐 + 03.2 re-verification** 才构成正式 closure。
- **D-04:** `FACT-01` 的满足方式继续固定为 **`FactionButton -> FactionPanel` 单场景 popup 详情链路**；战略态势不回塞 `MainHUD`，也不新增独立页面。

### Phase 3 复验权威口径
- **D-05:** `03-VERIFICATION.md` 应写成**正式 re-verification**，保留它作为 Phase 3 的主 verification 文档；不另起第二份“03.2 verification”平行真相文件。
- **D-06:** 新版 `03-VERIFICATION.md` 要按**完整 Phase 3 范围重跑**，重新核对四条 Phase 3 truths，而不是只补 `FACT-01` 小节；但结论中要突出 `FACT-01 blocker closed`。
- **D-07:** 对 downstream agents 与后续审计而言，**重跑后的 `03-VERIFICATION.md` 是 Phase 3 正式验收状态的唯一权威依据**；旧 summary 中的 `requirements-completed` 只能作为实现与历史证据，不能单独充当最终验收结论。

### Inserted-phase traceability 归一原则
- **D-08:** `REQUIREMENTS.md` 的 traceability 必须按**当前真实 ownership 全量归一**，不再保留 inserted phase 插入前的旧 phase 占位映射。
- **D-09:** requirement ownership 的判定原则固定为：**看“首次完整满足并正式验证”的 phase**，而不是看后续哪个 phase 继续扩展、复用或触碰相关功能。
- **D-10:** 对 `CARE-03` / `CARE-04` / `CARE-05` / `UI-03` 这类在 Phase 02.1 已首次完整满足的 requirement，`REQUIREMENTS.md` 继续把 owning phase 记为 **02.1**；Phase 3 的增强只写入其 verification / summary，不抢 ownership。
- **D-11:** 一旦 `REQUIREMENTS.md` 完成归一，`02.1-VERIFICATION.md` 中旧的 traceability drift 说明必须**删除或改写为已解决**，避免继续制造 stale audit noise。

### 文档收口边界
- **D-12:** 03.2 只有在以下三份文档同步收口后才算完成：`03-VERIFICATION.md`、`REQUIREMENTS.md`、`02.1-VERIFICATION.md`。
- **D-13:** `.planning/v1.0-v1.0-MILESTONE-AUDIT.md` 被视为**审计产物**，03.2 不手工重写它；当 verification 与 traceability 真相归一后，应通过后续 re-audit 生成新的 audit 文档来消除噪音。

### the agent's Discretion
- `03-VERIFICATION.md` 中 re-verification 头部声明、历史漂移说明、以及 `FACT-01 blocker closed` 的具体措辞。
- `REQUIREMENTS.md` traceability 表的精确 row wording，只要严格遵守单一 owning phase 与 inserted-phase 真实归属原则即可。
- `02.1-VERIFICATION.md` 中 stale note 的改写方式，只要最终表达“该漂移已由 03.2 归一解决”即可。
