## 1. 项目概述

### 1.1 项目一句话定义
一款基于 **190 年群雄割据剧本**、以 **单角色扮演** 为核心，融合 **《三国志10》式自由历史人物体验** 与 **《十字军之王》式士族门阀、联姻、子嗣、性格、派系政治** 的三国历史模拟单机游戏。

### 1.2 平台与技术
- 引擎：Godot
- 类型：单机 / 历史模拟 / 单角色策略 RPG / 势力博弈
- 目标阶段：先完成可验证核心体验的原型，再逐步扩展为完整项目

### 1.3 核心体验关键词
- 单角色
- 历史人物扮演
- 190 剧本
- 势力政治
- 士族门阀
- 联姻与家族
- 派系斗争
- 多结局历史演化

---

## 2. 设计目标

### 2.1 玩家体验目标
玩家应持续感受到：
- 自己是一个具体历史人物，而不是上帝视角
- 个人能力、关系、出身与政治环境共同决定命运
- 战争只是路径之一，仕途、婚姻、门阀网络同样重要
- 历史会发生，但可被玩家改写

### 2.2 设计成功标准
若首版实现成功，玩家应能在 15~30 分钟内感受到以下闭环：

**行动选择 → 人际反馈 → 功绩/名望变化 → 任命/机会变化 → 新目标出现**

---

## 3. 项目边界

### 3.1 首版必须坚持的边界
- 保持单角色视角
- 全身份共享同一底层逻辑
- 君主身份不能退化为传统全局 4X
- CK 特性必须做本地化改造，不直接照搬
- 优先做局部样本，不一开始做全国全人物完整内容

### 3.2 首版明确不做或延后做
- 完整全国地图漫游
- 超复杂战场即时表现
- 深度继承法细则
- 全量历史事件链
- 超细粒度经济系统
- 极深家谱/教育模拟

---

## 4. 核心玩法概述

### 4.1 主循环
**选择行动 → 消耗 AP/精力 → 获取成长/关系/功绩反馈 → 旬末/月末结算 → 获得新机会**

### 4.2 四条核心成长线
1. **个人成长线**：属性、技能、特性、名望
2. **仕途成长线**：功绩、官职、主公信任、任务权限
3. **家族成长线**：家族声望、联姻、子嗣、门第网络
4. **政治成长线**：派系站位、士族支持、势力影响力

### 4.3 长期目标线
- 忠臣名将
- 权臣/重臣
- 开国功臣
- 门阀领袖
- 乱世枭雄
- 隐退名士
- 家族延续者

## 5. 系统清单与优先级

## 5.1 P0 系统（必须先做）

### 时间推进系统
- 旬 / 月推进
- 统一结算
- 为所有系统提供时间钩子

### 角色系统
- 六维属性
- 身份/官职/所属势力
- AP / 精力 / 压力
- 名望 / 功绩 / 忠诚

### 行动系统
- 生成行动列表
- 校验条件
- 结算收益

### 关系系统
- 好感 / 信任 / 戒备 / 恩义
- 行动对关系的影响

## 5.2 P1 系统（闭环完成后接入）
- 任命与仕途系统
- 士族 / 家族系统
- 基础派系系统
- 任务与事件系统

## 5.3 P2 系统（增强阶段）
- 联姻系统
- 子嗣与接续机制
- 战争简化原型
- 历史事件分支

## 6. 核心系统摘要

### 6.1 角色系统
作用：承载玩家身份、属性成长、状态变化与系统权限差异。

关键字段：
- 属性
- 状态
- 身份
- 所属势力
- 家族/士族
- 派系归属
- 关系网络

### 6.2 士族门阀系统
作用：提供三国特有的出身、举荐、婚配、派系与政治资源修正。

三层结构：
- 士族/门阀
- 家族/支系
- 个人

### 6.3 家族 / 联姻 / 子嗣系统
作用：形成长期政治关系网与接续机制。

首版目标：
- 先做婚姻价值与姻亲关系
- 再做子嗣与继承扩展

### 6.4 性格 / 关系 / 压力系统
作用：让人物具备可感知的行为倾向与命运压力。

### 6.5 派系系统
作用：让势力内部政治可玩，影响任命、信任、升迁与叛离。

### 6.6 势力 / 内政 / 战争系统
作用：提供大局变化背景，并让玩家的个人成长与势力竞争相互嵌套。

---

## 7. 核心数值框架摘要

### 7.1 三资源模型
- **AP**：一旬能做几件事
- **精力**：本旬行动效果如何
- **压力**：中长期心理负担与恢复难度

### 7.2 关键可感知数值
- 六维属性
- 名望
- 功绩
- 忠诚
- 好感 / 信任 / 戒备
- 家族声望
- 派系影响力
- 城市金粮兵治安民心

### 7.3 数值设计原则
- 少核心值，多修正项
- 结果尽量可解释
- 允许逆风翻盘，但不允许无成本滚雪球

## 8. 时间结构与结算框架

### 8.1 时间单位
- 旬：主要行动单位
- 月：中结算单位
- 季：趋势评估单位
- 年：重大历史与角色变化单位

### 8.2 旬末结算
- 行动结果确认
- 关系变化
- 状态变化
- AI 行为推进

### 8.3 月末结算
- 任命与功绩评估
- 资源结算
- 婚姻/家族进度

### 8.4 季/年结算
- 派系重估
- 士族态度变化
- 重大历史事件推进

## 9. 身份玩法边界

### 9.1 支持身份
- 君主
- 武将
- 文臣
- 在野
- 隐士

### 9.2 统一规则
所有身份共享：
- 时间推进
- 行动点
- 状态系统
- 关系系统
- 历史事件参与逻辑

### 9.3 差异只体现在
- 行动菜单
- 权限范围
- 任务来源
- 事件池
- 结局方向

### 9.4 特别注意
君主身份必须仍受时间、关系、派系与执行链条限制。

## 13. AI 开发任务拆解建议

### 13.1 第一轮任务包
- 建立时间推进骨架
- 建立人物数据模型
- 建立行动菜单与基础结算
- 建立关系系统基础读写

### 13.2 第二轮任务包
- 接入功绩与任命
- 建立城市与势力基础数据
- 将行动结果接入月末反馈

### 13.3 第三轮任务包
- 接入士族/家族
- 接入派系倾向
- 加入基础事件系统

### 13.4 第四轮任务包
- 加入婚姻与姻亲关系
- 加入战争简化任务
- 加入局部历史事件验证

---

## 14. 验收标准

### 14.1 系统验收
- 时间推进稳定
- 行动消耗与收益可解释
- 关系变化可见且有效
- 功绩与任命形成闭环
- 士族/派系有感知差异

### 14.2 原型体验验收
- 玩家能连续玩 3~6 个旬不无聊
- 每旬至少有 2~3 个有效决策点
- 不同身份开局能看出差异
- 生活玩法不是纯装饰

### 14.3 架构验收
- 数据驱动可扩展
- UI 可替换
- 系统解耦基本成立

---

## 15. 风险与范围控制

### 15.1 高风险项
- 全身份深度差异化
- 复杂战争表现
- 全量历史人物与事件
- 深层家谱/继承模拟

### 15.2 控制原则
- 先做统一底层，再做身份差异
- 先做局部样本，再扩全国
- 先做事件接口，再扩大量内容
- 先做战果结算，再扩完整战役表现

## 17. 本阶段结论

本文件确认了本项目的开发 AI 工作方式：

1. **先把本文件当作总目录**
2. **再根据当前任务跳转到专项文档**
3. **按 P0 → P1 → P2 的顺序推进**
4. **保持单角色、历史模拟、势力博弈、士族门阀四个核心不偏移**

若后续进入实现阶段，开发 AI 应优先围绕：

**时间推进、行动系统、关系系统、任命闭环、士族修正**

这五项构建最小可玩原型。
