# Godot 原型开发拆解 v1

#项目设计 #开发拆解 #Godot #原型

> 目标：将现有设计成果拆解为 Godot 可执行的原型开发路径，明确最小可玩闭环、模块优先级、场景结构、数据驱动方式与阶段目标。

## 一、原型目标定义

### 1.1 本次原型的目标
不是直接做完整三国大作，而是先验证这款游戏最核心的体验是否成立：

> 玩家以单角色身份，在 190 年剧本下，通过旬制行动、人物交互、关系变化、任务与仕途成长，感受到“个人命运嵌入势力政治”的游戏体验。

### 1.2 原型必须回答的问题
1. 单角色主循环是否好玩？
2. 关系/名望/功绩是否能形成正反馈？
3. 士族门阀是否能带来可感知差异？
4. 全身份是否能建立在同一底层逻辑之上？
5. Godot 架构是否能支撑数据驱动扩展？

---

## 二、原型范围控制

## 2.1 原型必做内容（MVP）

### 核心循环必做
- 时间推进（旬/月）
- 玩家角色行动选择
- 人物属性与状态
- 人物关系变化
- 简单任务系统
- 功绩/名望/官职变化
- 势力归属与基础任命

### 特色系统必做
- 士族/家族基础数据
- 出身差异修正
- 基础联姻接口
- 基础派系归属与倾向显示

### 表现层必做
- 主界面
- 城市界面
- 人物详情界面
- 行动菜单
- 事件弹窗

## 2.2 原型暂缓内容
- 全地图自由移动
- 完整全国所有城池
- 复杂战场表现
- 完整子嗣教育链
- 深度继承法
- 大量历史事件链
- 全角色全家谱可视化

## 2.3 推荐原型样本规模
- 城市：**3~5 座**
- 势力：**2~3 个**
- 历史人物：**30~50 名**
- 士族：**5~8 个**
- 派系：每势力 **2~3 个主要派系**

建议优先选择：
- 洛阳 / 陈留 / 许县 / 邺城 / 濮阳 一类中原区域样本

---

## 三、最小可玩闭环

### 3.1 MVP 闭环定义

**旬开始 → 选择行动 → 消耗 AP/精力 → 获得关系/功绩/名望变化 → 旬末结算 → 月末任命/事件反馈 → 新目标生成**

### 3.2 玩家 15~30 分钟原型体验流程
1. 选择一名历史人物开局
2. 查看身份、所属势力、家族、士族、关系
3. 在城市中执行 2~4 个行动
4. 获得关系或属性成长
5. 触发一次事件或任务
6. 月末获得仕途反馈或新的政治机会

若这条链成立，原型就有价值。

---

## 四、Godot 原型架构建议

## 4.1 总体架构思路

建议采用：

**单主场景 + 数据管理器 + 多 UI 面板 + 事件驱动更新**

原因：
- 原型阶段不适合过早做复杂世界场景切换
- 更利于快速验证系统逻辑
- 便于后续逐步替换成正式场景结构

## 4.2 推荐基础结构

### 主控层
- `GameRoot`：游戏入口与主循环调度
- `TimeManager`：旬/月推进与统一结算
- `DataRepository`：读写人物、势力、城市、事件等数据
- `EventManager`：事件派发与监听
- `ActionResolver`：行动结算

### 业务层
- `CharacterSystem`
- `RelationSystem`
- `FactionSystem`
- `FamilyClanSystem`
- `CareerSystem`
- `ResourceSystem`
- `WarStubSystem`

### 表现层
- 主 HUD
- 城市界面
- 人物详情页
- 行动面板
- 事件弹窗
- 势力/派系信息面板

## 五、模块优先级

## P0：必须最先完成

### 5.1 时间与回合管理
功能：
- 旬推进
- 月末结算
- 统一广播事件

验证标准：
- 玩家能稳定推进时间
- 各系统能在旬末/月末正确刷新

### 5.2 角色数据系统
功能：
- 人物基础属性
- 身份、所属势力、官职
- AP / 精力 / 压力
- 名望 / 功绩

验证标准：
- 人物数据能被 UI 和行动系统读取与修改

### 5.3 行动系统
功能：
- 生成可执行行动列表
- 判断 AP 与条件
- 计算行动结果

验证标准：
- 玩家每旬至少可做 3~4 类不同动作

### 5.4 关系系统
功能：
- 好感 / 信任 / 戒备 等核心值
- 拜访、送礼、求教等行动带来的变化

验证标准：
- 关系变化对行动成功率或事件有明显影响

---

## P1：第一轮闭环完成后接入

### 5.5 功绩 / 官职 / 任命系统
功能：
- 玩家完成任务或行动获得功绩
- 月末进行任命评估
- 官职变化影响行动权限

### 5.6 士族 / 家族系统
功能：
- 人物绑定士族与家族
- 提供出身修正
- 提供联姻资格与举荐修正

### 5.7 基础派系系统
功能：
- 主要派系归属
- 派系支持/反对倾向
- 对任命与关系产生修正

---

## P2：原型增强阶段

### 5.8 任务与事件系统
功能：
- 支持主公委派、人物请求、家族事件
- 支持条件触发与分支结果

### 5.9 婚姻接口
功能：
- 显示可议婚对象
- 判定成功概率
- 建立姻亲关系

### 5.10 战争简化原型
功能：
- 只做“出征任务 + 战果结算”
- 暂不做完整战场

---

## 六、推荐场景与 UI 拆分

## 6.1 场景策略
原型阶段不建议先做开放地图漫游，建议以“界面驱动”原型起步。

### 推荐场景结构
1. `MainScene`：主场景
2. `CityPanel`：城市操作中心
3. `CharacterPanel`：角色信息页
4. `RelationPanel`：关系与社交页
5. `FactionPanel`：势力/派系页
6. `EventDialog`：事件弹窗
7. `WarResultPanel`：战果结算面板

### 这样做的优点
- 快速验证玩法
- UI 驱动更适合设计迭代
- 之后可逐步替换为更沉浸的地图与场景

## 七、数据驱动建议

## 7.1 原型阶段优先数据化的对象
- 人物
- 势力
- 城市
- 士族
- 家族
- 行动定义
- 官职定义
- 事件定义

## 7.2 数据字段优先级

### 人物数据最小字段
- id
- 姓名
- 性别
- 年龄
- 身份
- 所属势力
- 所在城市
- 六维属性
- AP / 精力 / 压力
- 名望 / 功绩 / 忠诚
- 家族 id
- 士族 id
- 派系 id
- 性格标签

### 城市数据最小字段
- id
- 名称
- 所属势力
- 金 / 粮 / 兵 / 治安 / 民心
- 在城人物列表

### 势力数据最小字段
- id
- 名称
- 君主 id
- 城池列表
- 主要派系列表
- 士族支持度

## 八、阶段开发计划

## 阶段 A：最小系统跑通（1~2 周）

目标：
- 实现旬推进
- 实现角色数据显示
- 实现基础行动菜单
- 实现 AP/精力/压力变化

完成标志：
- 玩家可在单城内完成一个完整旬循环

## 阶段 B：关系与成长闭环（1~2 周）

目标：
- 拜访/送礼/请教/训练
- 关系变化
- 属性成长
- 名望/功绩变化

完成标志：
- 玩家能感受到“做事后人物更强、关系更深”的反馈

## 阶段 C：仕途与势力闭环（1~2 周）

目标：
- 月末任命
- 势力归属
- 基础任务委派
- 官职权限差异

完成标志：
- 玩家能感受到“关系与功绩影响仕途”的闭环

## 阶段 D：士族与派系接入（1~2 周）

目标：
- 士族/家族修正接入
- 派系倾向接入
- 任命与联姻出现门第/派系影响

完成标志：
- 玩家能感知到“出身与政治网络”的存在

## 阶段 E：事件与战争简化接入（1~2 周）

目标：
- 任务事件分支
- 简化出征与战果结算
- 月/季事件驱动局势变化

完成标志：
- 原型具备基础历史模拟感

## 九、风险点与规避方案

### 9.1 风险：想一次做全身份完整玩法
规避：
- 底层统一，权限差异化
- 原型期先重点验证“在野/武将/文臣”三类

### 9.2 风险：过早做大地图与复杂战争
规避：
- 先做界面化战果系统
- 后续再扩到战役层

### 9.3 风险：历史数据量过大
规避：
- 先做局部样本区域
- 使用小规模历史人物集验证系统

### 9.4 风险：系统之间耦合太高
规避：
- 强制数据驱动
- 通过 TimeManager / EventManager 解耦

---

## 十、原型验收标准

若满足以下条件，则原型成功：

1. 玩家能完整跑完多个旬与至少 1 个月
2. 行动 → 关系 → 功绩 → 任命 的链条成立
3. 士族/家族出身差异可被玩家感知
4. 不同身份能基于同一套底层规则运行
5. UI 虽简化，但信息清晰可读
6. Godot 项目结构支持后续继续扩展

## 十二、本阶段结论

Godot 原型阶段应坚持以下原则：

1. **先闭环，后扩张**
2. **先数据，后表现**
3. **先局部样本，后全国大图**
4. **先验证单角色体验，后增加全身份差异深度**

这能最大限度降低项目失控风险，并尽快验证这款游戏真正的核心乐趣。
