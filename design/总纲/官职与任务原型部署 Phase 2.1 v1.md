# 官职与任务原型部署 Phase 2.1 v1

#项目设计 #GDD #仕途系统 #任务系统 #Phase2_1

> 文档定位：本文件是 Phase 2.1 的正式 GDD 章节稿，用于在进入 Phase 3「仕途、势力与可解释政治」前，先冻结“默认主角切换、月初领任务、月末结算、功绩驱动升官”的最小可玩仕途骨架。

---

## 1. 阶段概述

### 1.1 阶段定位
Phase 2.1 是正式进入 Phase 3 前的过渡阶段，用于先行完成“任务—功绩—官职”基础闭环，验证单角色仕途成长的最小可玩体验。

本阶段不追求完整政治博弈，而是优先搭建可复用的数据基础、结算逻辑与界面流程，为后续派系、推荐、反对、任命理由等系统提供稳定输入。

### 1.2 阶段目标
玩家以 **荀彧** 为默认角色，在月初选择一项主任务，通过当月行动推进任务进度，并在月末根据完成情况获得功绩、名望、信任变化；当功绩等条件达到阈值时，触发基础官职晋升。

### 1.3 核心体验陈述
玩家应在 10–20 分钟内明确感受到：

- 自己是势力中的一个具体人物，而非全局操控者；
- 每个月都要承担一项正式事务；
- 完成事务会积累功绩并影响仕途；
- 官职提升不是纯文本，而会带来新的任务层级与身份反馈。

---

## 2. 设计目标

### 2.1 体验目标
1. 建立明确的月度目标感：玩家每月都有“要完成的事”。
2. 建立基础仕途反馈：任务结果会进入月末评价。
3. 建立基础成长驱动力：功绩推动升官，升官解锁新事务。
4. 保持单角色边界：即使升官，也不进入传统君主全局经营模式。

### 2.2 系统目标
1. 完成默认角色切换：曹操 → 荀彧。
2. 建立最小可用官职体系。
3. 建立最小可用任务体系。
4. 建立月初选任务、月末结算、升官判定完整流程。
5. 为 Phase 3 预留可扩展字段：推荐、支持、反对、派系标签、职位权限。

### 3.1 本阶段包含内容
- 默认主角替换为荀彧
- 官职数据与基础晋升逻辑

上述内容属于 Phase 3 正式范围。

## 4. 角色设定调整

### 4.1 默认角色切换

#### 设计决定
将原默认角色 **曹操** 调整为 **荀彧**。

#### 调整原因
1. 曹操天然具备君主属性，容易破坏“单角色执行”边界。
2. 荀彧更适合承接文臣仕途成长、任务委派、任官评价等系统。
3. 荀彧可自然连接后续中枢政治、文臣关系网、举荐与派系机制。
4. 更利于突出“个人命运嵌入势力政治”的项目核心价值。

### 4.2 荀彧原型身份设定
- 所属势力：曹操
- 初始身份：低中阶幕僚 / 从事类官员
- 初始定位：政务型、组织型、协调型文臣
- 初始优势：
  - 智略较高
  - 政务适性较高
  - 初始信任较好
  - 可承接中枢型任务
- 初始短板：
  - 军事执行能力一般
  - 需要通过持续事务积累功绩
  - 尚未进入高度可解释政治博弈层

## 5. 核心玩法循环

### 5.1 月度循环结构
本阶段采用固定的“月初—月内—月末”三段式循环：

#### 月初
- 自动弹出任务选择界面
- 玩家从候选任务中选择 1 个主任务

#### 月内
- 玩家执行常规行动推进任务进度
- 可获得阶段性反馈

#### 月末
- 结算任务完成情况
- 结算功绩、名望、信任变化
- 判定是否满足升官条件
- 更新下月任务池

### 5.2 循环目标
通过稳定、重复、可解释的月度节奏，让玩家逐步建立对“事务 → 功绩 → 升官”的理解，为后续 Phase 3 政治系统提供认知基础。

## 6. 官职系统设计

### 6.1 设计原则
本阶段官职系统以“**功能分层**”为主，不追求完整历史官制还原。

官职在原型中的作用包括：

1. 表示玩家当前仕途阶段；
2. 控制任务池解锁；
3. 提供月末升迁目标；
4. 形成身份反馈与成长动力。

### 6.2 官职层级设计
建议采用 4 级原型官职结构：

#### 1）白身 / 门客
- 功能定位：仕途起点或特殊开局身份
- 任务类型：基础杂务、低风险文书、私人协助
- 政治含义：未正式进入考核体系

#### 2）从事 / 属吏
- 功能定位：正式进入势力政务链条
- 任务类型：基础政务、后勤、文书、协调
- 政治含义：被上级正式观察与考评

#### 3）主簿级辅官
- 功能定位：承担更稳定、更核心的内政事务
- 任务类型：资源调度、士族安抚、人才举荐、行政整顿
- 政治含义：开始具备一定政治可见度

#### 4）中枢幕僚级
- 功能定位：接触更高层政务与统筹事务
- 任务类型：中枢调度、关键任命建议、重要协调任务
- 政治含义：为下一阶段政治博弈做准备

### 6.3 官职效果
本阶段官职至少产生以下影响：

#### 1）任务池差异
不同官职看到的任务候选不同。

#### 2）收益差异
高层级任务通常带来更高功绩收益与更重要评价。

#### 3）文本待遇差异
NPC 称呼、月末评语、界面描述随官职变化。

#### 4）后续扩展接口
后续 Phase 3 可基于官职增加：
- 推荐资格
- 任命竞争
- 查看更多势力信息
- 接触更高层人物

## 7. 任务系统设计

### 7.1 设计目标
任务系统用于将玩家的日常行动转化为可被月末结算读取的正式贡献记录，是本阶段最重要的输入系统。

### 7.2 任务来源
本阶段仅采用单一主来源：

#### 上级指派任务
由曹操或直属上级在月初发出。

#### 设计原因
- 逻辑清楚，易于解释
- 功绩归属明确
- 有利于月末评价闭环
- 降低原型复杂度

### 7.3 任务选择规则
- 每月开始时自动出现任务选择界面
- 同时提供 2–3 个候选任务
- 玩家只能选择 1 个作为当月主任务
- 所选任务在当月内保持有效
- 若当月结束前未达成条件，则按失败 / 未完成处理

### 7.4 任务类型原型
建议优先部署以下 4 类任务模板：

#### 1）整理军粮
- 类型：内政 / 后勤
- 内容：核对军粮、协助调拨、整理储备
- 奖励倾向：稳定功绩，少量名望

#### 2）安抚士族
- 类型：政治 / 关系
- 内容：拜访地方士族，稳定合作意愿
- 奖励倾向：中等功绩，少量名望与关系成长

#### 3）举荐人才
- 类型：政务 / 人事
- 内容：接触特定人物，完成举荐链条
- 奖励倾向：较高功绩，较高信任成长

#### 4）整顿文书
- 类型：行政 / 文案
- 内容：完成一系列低风险政务动作
- 奖励倾向：收益稳定，适合作为保底任务

### 7.5 任务完成等级
任务结果分为三档：

#### 成功
完成核心条件，获得标准奖励。

#### 优秀完成
完成核心条件且表现优于预期，获得额外奖励。

#### 失败 / 未完成
未达到核心条件，奖励减少或无奖励，并可能产生轻微负反馈。

### 7.6 任务字段建议
每个任务原型至少包含以下数据：

- 任务 ID
- 任务名称
- 发布人
- 所属类型
- 官职需求
- 任务描述
- 完成条件
- 执行期限
- 标准奖励
- 优秀奖励
- 失败结果
- 后续解锁标签（预留）

## 8. 功绩与评价系统

### 8.1 功绩定位
功绩是本阶段最核心的仕途成长指标，用于衡量玩家对势力的正式贡献，并作为升官判定的主要依据。

### 8.2 辅助指标
除功绩外，本阶段保留两个轻量辅助指标：

#### 名望
反映社会层面的知名度与口碑，当前主要用于丰富反馈与后续扩展。

#### 信任
反映主君或上级对玩家的认可，当前主要用于辅助文本反馈与未来扩展任命逻辑。

### 8.3 数值变化来源
- 任务成功：功绩增加
- 优秀完成：额外功绩或名望 / 信任增加
- 失败：功绩不变或小幅受损
- 特定任务：偏向增加名望或信任

### 8.4 原型建议
为控制复杂度，本阶段升官判定以 **功绩值为主**，名望、信任仅作为辅助展示，不作为硬门槛。

## 9. 月末结算系统

### 9.1 结算目标
月末结算负责把“玩家本月行为”转化为“仕途结果”，并以清晰反馈告知玩家自己为什么升官、为什么没有升官。

### 9.2 结算顺序
建议固定为：

1. 主任务完成情况判定
2. 奖励结算（功绩 / 名望 / 信任）
3. 升官判定
4. 官职变化反馈
5. 下月任务池刷新

### 9.3 升官规则
本阶段使用简单阈值式升官逻辑。

#### 示例阈值
- 白身 → 从事：10 功绩
- 从事 → 主簿级：25 功绩
- 主簿级 → 中枢幕僚级：45 功绩

#### 规则说明
- 满足阈值即具备升迁资格
- 若当月任务失败，则本月不触发升迁
- 已达到当前阶段最高官职时，仅提示“仕途评价上升，但暂无更高任命”

### 9.4 月末反馈要求
月末报告必须做到基础可解释，至少展示：

- 本月所选任务
- 任务结果（成功 / 优秀 / 失败）
- 功绩变化
- 名望变化
- 信任变化
- 当前官职
- 是否升官
- 若未升官，显示距离下一级所需功绩

#### 示例文本
**本月仕途结算**
- 所办事务：整理军粮
- 结果：优秀完成
- 功绩 +10
- 名望 +1
- 曹操信任 +1

**官职判定**
- 当前功绩：26
- 已达到升迁标准
- 官职由“从事”升为“主簿级辅官”

**影响**
- 下月可接更高阶政务任务
- 仕途评价提高

## 10. 界面与交互设计

### 10.1 月初任务选择界面

#### 功能目标
让玩家在每个月开始时立即理解“本月重点”。

#### 显示内容
每个任务卡片建议展示：
- 任务名称
- 发布人
- 简要描述
- 任务类型
- 难度
- 推荐方向
- 预计奖励

#### 交互要求
- 自动弹出
- 可浏览多个候选任务
- 只能选定一个
- 选定后明确标记为“本月主任务”

### 10.2 月内任务状态显示
界面常驻显示：
- 当前主任务名称
- 当前任务进度
- 剩余时间
- 关键完成条件提示

### 10.3 月末结算界面

#### 必显信息
- 任务结果
- 功绩 / 名望 / 信任变化
- 官职是否变化
- 下一级目标

#### 设计要求
- 一屏内读懂
- 因果清晰
- 不使用复杂政治演算说明
- 重点突出“努力是否带来仕途推进”

## 13. 成功标准

本阶段完成后，以下内容必须为真：

1. 默认角色已切换为荀彧，且角色定位适合承接仕途型玩法。
2. 每月开始时，玩家会收到至少 2–3 个候选任务，并能选择其中 1 个作为当月主任务。
3. 玩家可以在月内推进任务，并在月末看到明确的任务完成结果。
4. 任务结果会影响功绩值，功绩达到阈值后可触发基础升官。
5. 官职变化至少会影响任务池与身份反馈文本。
6. 整个流程形成稳定的“月初领任务—月内推进—月末升官判定”闭环。

## 14. 风险与控制

### 14.1 主要风险

#### 风险一：任务过少导致重复感强
控制方式：任务模板虽然少，但尽量覆盖后勤、政务、关系、人事四类。

#### 风险二：升官太快或太慢
控制方式：升迁阈值按 15–30 分钟体验至少触发 1 次升官来校准。

#### 风险三：官职变化感知不足
控制方式：至少保证任务池、称呼、月末评语三项有明确差异。

#### 风险四：荀彧体验仍像“代办任务”
控制方式：任务描述与月末反馈要强化“仕途成长”语义，而非纯跑腿语义。

## 16. 测试要点

### 16.1 功能测试
- 月初是否稳定弹出任务选择
- 是否只能选择一个主任务
- 月内任务进度是否正确更新
- 月末是否正确结算任务结果
- 功绩是否正确累计
- 升官是否按阈值正确触发

### 16.2 体验测试
- 玩家是否理解本月要做什么
- 玩家是否理解为什么升官或未升官
- 官职变化是否有明显感知
- 荀彧是否比曹操更符合单角色仕途体验

### 16.3 平衡测试
- 任务奖励是否足以驱动成长
- 升迁节奏是否适中
- 保底任务是否过强
- 高收益任务是否值得承担风险

---

## 17. 本章结论
Phase 2.1 的作用不是扩展复杂内容，而是将项目从“角色行动原型”推进到“**可验证的仕途成长原型**”。

它必须先完成最小可用的任务、功绩与官职闭环，再进入 Phase 3 的政治解释层，否则后续派系与任命系统会缺乏稳定支撑。
