# 项目总设计方案 v1

#项目设计 #三国 #Godot

> 说明：本文件为已对齐的《项目总设计方案 v1》，存档并作为后续 GDD 与开发拆解的设计文档起点。

## 一、项目定位

### 1.1 项目一句话定义
一款基于 **190年群雄割据剧本**、以 **单角色扮演** 为核心，融合 **《三国志10》的自由武将玩法** 与 **《十字军之王》的士族门阀/家族关系/派系政治** 的三国历史模拟单机游戏。

### 1.2 核心体验目标
玩家不是在玩一个纯势力经营器，而是在玩一个 **“身处乱世中的具体历史人物”**：

- 作为 **个人** 生存、成长、结交、仕官、立功、成婚、生子、积累声望
- 作为 **家族成员** 受到出身、门第、婚姻网络、士族评价的影响
- 作为 **势力成员** 卷入主公、同僚、派系、功劳分配、权力斗争
- 作为 **时代参与者** 推动或改变 190 年后的历史进程

### 1.3 核心卖点
1. **单角色全身份体验**：可扮演君主、武将、文臣、在野者、隐士等历史人物，但始终保持“单角色视角”。
2. **士族门阀系统**：更贴合三国时代的“士族—家族—个人”三层结构。
3. **关系驱动的历史模拟**：婚姻、子嗣、门第、恩怨、师友、派系，比单纯数值更影响仕途与命运。
4. **势力内外双重博弈**：处理内部派系、功劳分配、主公信任、同僚竞争。
5. **历史还原 + 多结局演化**：以历史为骨架，但允许玩家干预关键节点。

## 二、设计原则

### 2.1 第一原则：始终坚持“单角色”

即使玩家扮演君主，也不是直接变成传统全局 SLG。

**君主身份的设计约束**：
- 君主仍是“一个人”，有体力、时间、关系、性格、声望限制
- 不能无限制同时处理所有事务
- 许多决策需要通过臣属执行
- 君主也会受派系、家族、婚姻、继承问题掣肘

**结论：** 君主身份是“权力更大”，不是“玩法换成全势力上帝视角”。

### 2.2 第二原则：CK 特性必须本地化

不是照搬欧洲封建制，而是转化为三国语境下的系统。

**对应关系**：
- CK家族 → **士族/门阀 + 家族**
- CK婚姻外交 → **联姻、门第结合、政治婚配**
- CK继承 → **子嗣、宗族延续、家业继承**
- CK性格 → **人物性格、行为倾向、关系偏好**
- CK派系 → **势力内部派系、同乡集团、门生故吏网络**
- CK封臣忠诚 → **部曲、门客、属吏、将领忠诚**

### 2.3 第三原则：先保证可玩，再追求复杂

复杂系统必须为玩法服务，不能变成纯文本模拟器。

## 三、游戏核心循环

### 3.1 主循环
**行动 → 交互 → 成长 → 博弈 → 结果反馈 → 新目标**

具体表现为：

1. **行动**
   - 城市移动
   - 外出拜访
   - 执行任务
   - 出征/巡察/游历/外交

2. **交互**
   - 与 NPC 对话
   - 接受委任
   - 招揽、结交、拜师、结义、求婚
   - 舌战、单挑、说服、拉拢

3. **成长**
   - 属性成长
   - 技能学习
   - 官职提升
   - 名望/恶名变化
   - 家族地位变化

4. **博弈**
   - 势力战争
   - 派系斗争
   - 士族联姻
   - 忠诚管理
   - 功劳与任命竞争

5. **反馈**
   - 获得封赏/失宠
   - 婚姻成功/失败
   - 派系壮大/遭打压
   - 历史事件分歧
   - 人生结局变化

## 四、玩家身份体系

### 4.1 身份范围
首发设计支持：君主、武将、文臣、在野、隐士/游侠类人物。

### 4.2 各身份核心玩法差异

**君主**：任命、战略、外交、赏罚、继承安排；重点在“权力管理”。

**武将**：战功、统兵、结交、受封、自立；强调战争与个人武勇成长。

**文臣**：内政、外交、谋略、举荐、派系经营；强调政治影响力。

**在野**：游历、结识、待价而沽、自创势力前置；最自由，适合乱世崛起。

**隐士**：不主动入仕，可通过事件被征辟；偏特殊路线和另类结局。

## 五、核心系统框架

### 5.1 角色系统

**基础属性（建议）**：统率、武力、智略、政务、魅力、名望。

**衍生属性**：忠诚、野心、健康、压力/心境、声誉、门第声望。

**技能体系**：
- 通用技能：骑战、枪战、军略、辩才、政务、察人等
- 专属特技：名士、豪族领袖、善战、善辩、识人、仁德、酷吏等

### 5.2 士族门阀系统（本作核心特色）

三层结构：
1. **士族/门阀**（如弘农杨氏、颍川荀氏、汝南袁氏）
2. **家族**（玩家所属支系）
3. **个人**（能力、性格、名望、成就决定能否超越出身）

士族提供的影响：初始人脉、仕官难度修正、初始资源/教育修正、婚配对象质量、派系归属倾向、主公信任或戒备。

设计目标：让“寒门逆袭”和“世家经营”都成为可玩路线。

### 5.3 家族 / 联姻 / 子嗣系统

联姻作用：政治结盟、缓和仇怨、打开仕途、提升门第、巩固派系。

子嗣作用：血脉延续、继承家业/官位影响、后期接班、家族关系网络扩张。

落地原则：不做过度复杂的 CK 式繁殖模拟，重点做 **婚配价值、子嗣继承、家族延续感**。

### 5.4 性格 / 关系系统

性格标签（建议）示例：仁厚、果决、多疑、傲慢、谨慎、好名、重情、残酷、激进、保守。

性格作用：影响 AI 行为、关系变化、事件选项、派系兼容性、压力增长方式。

关系类型：亲近、友好、中立、厌恶、仇敌、亲族、姻亲、师徒、结义、君臣、门生故吏。

### 5.5 派系系统

派系来源：同乡、同族、同门、姻亲、功臣集团、文臣集团/武将集团、主战/主和、扶嫡/扶庶。

派系玩法价值：影响任命、进言成功率、忠诚、叛变与投靠、继承与内乱、君主对臣下的平衡策略。

核心原则：派系不是独立地图层，而是**嵌入势力内部政治**。

### 5.6 势力与战争系统

势力构成：城池、武将/文臣、兵力、粮草、金钱、士气、派系结构、门阀支持度。

战争层级：
1. 个人层（单挑、战术指令、武将表现）
2. 部队层（编组、兵种、阵型、计略）
3. 战役层（出征、围城、援军、断粮、撤退）

设计建议：首版优先战略层推进、局部战斗/战役界面，强调玩家角色在战中的作用。

### 5.7 内政与仕途系统

内政行为：征兵、巡察、训练、发展农业/商业、治安、人才搜寻、外交出使。

仕途推进：功绩、官职、信任、士族加成、派系支持、婚姻关系、历史事件表现。

重点：升迁不能只靠数值，还要看关系和政治环境。

### 5.8 历史事件系统

事件类型：历史主线事件、人际事件、家族事件、派系事件、战争事件、婚姻/继承事件。

触发原则：历史时间 + 条件判断 + 人物状态 + 势力格局。

结果结构：历史照常发生 / 玩家推动提前或延后 / 玩家改变归属或结果 / 玩家完全破坏原历史链。

## 六、项目特色定义

**6.1 不是普通三国 SLG**：你不是俯瞰天下，而是在天下中谋生、谋权、谋名。

**6.2 不是纯 RPG**：被卷入真实的政治网络。

**6.3 不是纯 CK 换皮**：核心语境是三国士人政治、门阀关系、军政博弈。

## 七、建议的首版范围控制

为可落地，首版 v0.1 建议包含：

### 必做
- 190 剧本
- 单角色地图行动
- 城市/势力/NPC 基础交互
- 仕官/在野/君主基本流程
- 属性、技能、关系
- 士族门阀基础框架
- 联姻与亲族关系
- 派系基础框架
- 任命/任务/功绩/官职
- 简化战争与出征
- 关键历史事件链

### 延后
- 复杂继承法
- 大量随机支线文本
- 极复杂子嗣教育
- 全地图超细粒度经济
- 过深的后宫/宫廷阴谋
- 超复杂实时战场

## 八、项目风险与边界

### 8.1 最大风险
1. **全身份导致系统爆炸**
2. **君主玩法滑向传统全局 SLG**
3. **CK 特性过多导致开发失控**
4. **历史人物与门阀数据量巨大**

### 8.2 控制办法
1. 先统一底层逻辑，再做身份差异
2. 君主也必须遵守单角色行动约束
3. 士族/派系先做“可感知”，后做“深模拟”
4. 首版只做核心名门与关键人物网络

## 九、当前对齐后的产品定义

> **定义**：以190年乱世为起点，玩家扮演任意历史人物，在单角色视角下，通过个人成长、仕途经营、士族门阀网络、婚姻子嗣关系与势力派系斗争，参与并改写三国历史的角色历史模拟游戏。

## 十、下一步建议

如果认可本方案，下一步建议按顺序：

1. **先做《核心系统详细设计》**（角色、士族、派系、婚姻、身份边界）
2. **再做《主循环与数值骨架》**（时间结构、行动力、功绩/忠诚/关系逻辑、势力运转）
3. **最后做《GDD 框架 + Godot 开发拆解》**
