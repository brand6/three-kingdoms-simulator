# Phase 3 仕途、势力与可解释政治 详细规划 v1

#项目设计 #GDD #Phase3 #仕途系统 #势力系统 #派系系统 #任命系统

> 文档定位：本文件是 Phase 3 的正式 GDD 章节稿，用于在 Phase 2.1 已冻结的“月初领任务、月内推进、月末结算、功绩驱动升官”闭环之上，进一步建立单角色视角下的可解释政治层，明确仕途推进、势力信息可见性、推荐/反对链、派系支持、任命解释、官职权限变化与月末政治结算规则。

---

## 1. 阶段概述

### 1.1 阶段定位
Phase 3 是从“任务驱动升官原型”进入“可解释政治原型”的关键阶段。

本阶段不再把月末结果只当作数值阈值判定，而是要让玩家理解：自己为何被推荐、为何被反对、为何得到派系支持、为何错失任命，以及这些结果如何反过来影响下月的仕途机会。

### 1.2 阶段目标
玩家仍以单角色身份在势力内部行动，但每月会逐步感受到：

- 自己承担的事务来自不同政治来源；
- 努力结果会进入推荐、反对与派系评价；
- 任命不是黑箱，而是可回溯到资格、关系、支持、阻力与空缺；
- 官职变化会改变自己能做什么、能看到什么、会承担什么政治后果。

### 1.3 核心体验陈述
玩家应在 15–30 分钟内明确感受到：

- 自己是势力中的一个具体人物，而非俯瞰全局的人事调度者；
- 势力内部存在有限但可读的政治结构；
- 月末结果不是随机发落，而是由可见关系链和政治环境共同推动；
- 升官、落选、遭压制、错失机会都具有可解释因果。

---

## 2. 设计目标

### 2.1 体验目标
1. 让玩家感受到“仕途推进嵌在势力政治中”，而不是孤立的任务刷分。
2. 让玩家能读懂主要政治支持者、主要阻力来源与当前派系环境。
3. 让任命结果具备基础可解释性，而不是只显示成败。
4. 让官职变化真正改变权限、待遇反馈与下月政治空间。
5. 保持单角色视角，坚决不滑向君主全局 SLG。

### 2.2 系统目标
1. 扩展月任务来源，使其至少来自两类可解释政治入口。
2. 建立推荐、反对、派系支持与任命原因的统一解释结构。
3. 建立最小可用的候选竞争与月末政治结算流程。
4. 建立官职与权限、信息可见性、待遇文本之间的稳定映射。
5. 让人物卡、势力页、月末报告复用同一套政治解释数据。

---

## 3. 范围定义

### 3.1 本阶段包含内容
- 在 Phase 2.1 月度闭环上扩展政治解释层
- 势力信息可见性分层
- 月任务来源扩展为至少两类政治来源
- 推荐 / 反对来源累积与月末写回
- 派系支持与有限候选竞争
- 任命结果原因分解
- 官职权限、发言资格、待遇反馈变化
- 月末政治结算与下月建议反馈
- 人物政治摘要、势力政治总览、月末政治报告

### 3.2 本阶段不包含内容
- 君主模式下的全势力内政总控
- 城池级宏观生产调度与外交沙盘
- 自定义派系创建与复杂派系议程系统
- 完整朝廷官制模拟
- 大而全的全国政治地图
- 家族婚姻与继承深度联动

上述内容属于更后续的扩展范围，不在本阶段内解决。

## 4. 单角色边界与反 SLG 约束

### 4.1 核心约束
Phase 3 虽然引入势力、任命、派系与人事竞争，但本质上仍是“玩家作为一个人进入政治环境”，而不是“玩家接管整个势力系统”。

### 4.2 具体边界
1. 玩家只能看到与自己官职、关系、位置相关的政治信息，而不是默认掌握全部后台数据。
2. 玩家只能通过接任务、拜访、进言、建立关系、争取支持等个人行为影响结果。
3. 玩家不能像传统 SLG 君主一样直接批量任命、批量调兵、批量下达人事指令。
4. 即使玩家扮演高位角色，权限提升也表现为“接触面变大、话语权更强、选择更多”，而不是切换到全局上帝模式。

### 4.3 设计判断标准
若某个交互让玩家像在操作势力仪表盘而不是在扮演具体人物，则该设计不应进入 Phase 3。

## 5. 核心玩法循环

### 5.1 月度循环结构
Phase 3 继续沿用固定的“月初—月内—月末”三段式循环，但每一段都增加政治解释层。

#### 月初
- 刷新候选任务
- 展示任务来源与相关人物
- 展示当前主要支持 / 阻力摘要
- 玩家选择 1 个当月主任务

#### 月内
- 玩家通过既有行动推进主任务
- 拜访、交涉、执行政务、完成委托会累积政治影响
- 官职权限决定部分行动是否可见或可执行

#### 月末
- 结算任务结果
- 写回功绩、名望、信任与政治支持变化
- 进行候选资格与任命竞争判定
- 输出可解释的政治结论
- 将结果转化为下月机会、阻力或待遇变化

### 5.2 循环目标
通过稳定、重复、可回溯的月度结算，让玩家逐步形成以下理解：

1. 事务表现会进入政治评价。
2. 政治评价会影响任命机会。
3. 任命结果会改变权限与政治处境。
4. 政治处境又会影响下一月能接到什么事、被谁支持、被谁阻拦。

## 6. 势力信息可见性设计

### 6.1 设计原则
本阶段不是让玩家“拥有全部势力数据”，而是让玩家“拥有足够理解政治环境的信息”。

### 6.2 玩家可见的势力信息
玩家在 Phase 3 中应可查看以下可解释信息：

#### 1）势力核心结构
- 主君
- 核心高层官员
- 玩家所在城市与所属行政链
- 当前主要政务重心或军事压力

#### 2）主要派系块
- 派系名称或倾向标签
- 派系核心人物
- 派系当前立场摘要
- 派系对玩家的大致倾向

#### 3）玩家政治处境
- 主要推荐人
- 主要反对者
- 当前资格短板
- 当前可争取机会

### 6.3 不直接暴露的信息
为保持单角色视角，本阶段默认不直接展示：

- 全势力所有隐藏权重
- 全官员详细后台忠诚数值
- 全职位实时竞争分数排行榜
- 势力所有资源的全局精确调度面板

### 6.4 展示方式要求
信息展示必须偏摘要化、叙事化、因果化，例如：

- “颍川士人多认可你的政务能力”
- “主战一派对你持保留态度”
- “主簿一职暂无空缺，且有旧吏优先竞争”

而不是直接堆出大量后台数字表。

## 7. 任务来源与政治入口设计

### 7.1 设计目标
Phase 2.1 的任务只解决“有事可做”；Phase 3 的任务必须开始解决“这些事是谁递到你面前的”。

### 7.2 任务来源扩展原则
本阶段继续保持“每月只接 1 个主任务”的稳定节奏，但任务来源不再只有上级指派。

### 7.3 原型任务来源
建议至少部署以下两类来源：

#### 1）势力指派任务
- 来源：主君、直属上级、官署链条
- 特征：正式、稳定、功绩归属清晰
- 政治意义：反映你是否仍在制度性仕途通道中

#### 2）人际 / 士族请求任务
- 来源：同僚、门生故吏、士族代表、关系人物
- 特征：带有人情与关系网络色彩
- 政治意义：反映你是否正在被某一政治网络吸纳、试探或利用

### 7.4 候选任务显示要求
每个候选任务卡片建议额外展示：

- 任务来源类型
- 发布者 / 请求方
- 关联派系或关系网络
- 接下该任务的潜在政治收益
- 潜在政治风险或压力点

### 7.5 范围控制
本阶段不引入并行主线、海量支线、任务日志膨胀或多主任务制度。

核心目标不是“任务更多”，而是“任务来源更有政治含义”。

## 8. 推荐与反对链设计

### 8.1 设计原则
推荐与反对不是独立数值小游戏，而是既有关系、信任、功绩、官职与任务结果在政治层的延伸表达。

### 8.2 推荐来源
推荐可由以下因素产生：

1. 长期关系良好
2. 上级信任提高
3. 月任务优秀完成
4. 官职资格匹配
5. 士族 / 同乡 / 同门网络加成
6. 某派系认为你有助于其当前目标

### 8.3 反对来源
反对可由以下因素产生：

1. 与关键人物关系恶化
2. 月任务失败或表现平庸
3. 与既有候选人形成竞争
4. 身份、资历或门第不被接受
5. 某派系认为你会削弱其影响力
6. 曾经在月末结算中留下负面政治记录

### 8.4 推荐 / 反对链的玩法价值
玩家需要逐步理解：

- 谁在推自己
- 谁在压自己
- 这些立场从何而来
- 自己还能通过什么行动去修正

### 8.5 原因记录要求
每次推荐或反对变化，都应至少留下最小原因行，例如：

- “因你本月安抚士族表现出色，荀攸增加了对你的举荐意愿”
- “因你与旧任主簿关系紧张，对方开始阻挠你的升迁”
- “因你出身与该职所需网络不匹配，部分士人不愿支持”

这些原因行必须可复用于人物卡、势力页与月末报告。

## 9. 派系支持与任命竞争设计

### 9.1 设计目标
派系不是装饰性标签，而是任命结果的重要修正来源；但在 Phase 3 中，其实现必须保持轻量、可读、可控。

### 9.2 派系支持的最小表达
本阶段派系支持先采用“倾向 + 修正 + 原因说明”结构。

#### 倾向示例
- 支持
- 保留
- 反对
- 竞争性压制

#### 影响示例
- 增加任命认可度
- 增加推荐出现概率
- 提升落选风险说明中的政治阻力
- 改变下月任务来源偏向

### 9.3 多候选竞争范围
本阶段允许玩家与 1–2 名 AI 候选人争夺同一职位，但竞争流程必须保持克制。

#### 竞争比较维度
- 功绩是否达标
- 当前信任与关系基础
- 推荐人强度
- 反对压力
- 派系倾向
- 官职资历或职位资格

### 9.4 竞争结果类型
任命竞争至少应输出以下结果之一：

- 任命成功
- 因资格不足未进入候选
- 因推荐不足未被正式提名
- 因反对 / 压制而落选
- 因空缺不存在而错失机会
- 因竞争者优势更强而败选

### 9.5 范围控制
本阶段不做：

- 复杂派系议程投票
- 派系资源运营
- 玩家自建派系与公开站队系统
- 大规模多职位同步竞争

## 10. 任命解释系统设计

### 10.1 设计目标
任命结果必须从“只给结果”升级为“给结果 + 给原因 + 给下月改进方向”。

### 10.2 任命解释的标准分层
建议将任命解释拆为以下五层：

#### 1）资格层
- 是否满足官职、资历、功绩等基础门槛

#### 2）制度层
- 该职位是否有空缺
- 是否已处于冻结或观望状态

#### 3）推荐层
- 是否有人愿意明确举荐
- 推荐强度是否足以推动进入候选

#### 4）阻力层
- 是否存在关键反对者
- 是否遭遇派系压制或旧人留任逻辑

#### 5）竞争层
- 与其他候选人相比，自己输在何处

### 10.3 任命解释输出要求
任命界面或月末报告中，应至少能读出：

- 你为什么成功
- 你为什么失败
- 失败属于哪一类
- 下月最值得改进的方向是什么

### 10.4 示例表达
#### 成功示例
- “你在本月整顿文书事务中表现优秀，获得上级正式举荐；虽有旧吏保留意见，但主战派并未反对，且主簿职位正好空缺，因此本月任命成立。”

#### 失败示例
- “你已具备功绩门槛，但本月未形成足够明确的推荐链，且该职位暂由旧人留任，因此本次未被列入正式候选。”

## 11. 官职权限变化设计

### 11.1 设计原则
官职变化在 Phase 3 中不能再只是任务池与称呼变化，而必须开始改变玩家可见范围、可参与事务和失败代价。

### 11.2 官职变化至少影响以下内容

#### 1）任务可见性
高一级官职可接触更核心、更敏感或更接近中枢的任务。

#### 2）信息可见性
高一级官职可查看更多势力摘要、职位动态与派系倾向。

#### 3）发言资格
部分进言、举荐、议事、协调类行动需要达到特定官职层级。

#### 4）候选资格
某些职位不只是功绩门槛，还要求已有特定层级的履历。

#### 5）待遇反馈
升官或落选会改变 NPC 称呼、月报文案、政治评价语气。

#### 6）政治后果
高官职阶段的失败不再只是“没升官”，还可能造成信任下滑、下月被边缘化或机会变差。

### 11.3 权限变化的落地要求
所有权限差异都应接入既有单角色 HUD、任务流程与行动目录，不单独创造“君主内政面板”或“全局人事界面”。

## 12. 月末政治结算系统

### 12.1 结算目标
月末政治结算负责把“本月事务结果”转化为“仕途位置变化”，并把隐藏在后台的支持、阻力、资格与竞争结果转译为玩家可理解的政治结论。

### 12.2 结算顺序
建议固定为：

1. 主任务完成情况判定
2. 功绩 / 名望 / 信任写回
3. 推荐 / 反对 / 派系支持快照写回
4. 候选资格判定
5. 空缺与竞争判定
6. 任命结果生成
7. 下月政治机会与风险刷新

### 12.3 结算输出内容
月末政治结算至少应输出：

- 本月所选任务及来源
- 任务结果与关键贡献
- 功绩 / 名望 / 信任变化
- 主要推荐来源
- 主要阻力来源
- 派系态度摘要
- 任命结果
- 若失败，失败类别与原因分解
- 若成功，新增权限与新职责提示
- 下月建议方向

### 12.4 下月建议示例
- “若想争取主簿任命，应优先补强与荀攸关系并完成一次高可见度政务任务。”
- “当前主战派对你仍有保留，下月不宜主动争取军政混合职位。”
- “职位暂无空缺，继续积累功绩并维持推荐链更有价值。”

## 13. 界面与交互设计

### 13.1 月初任务选择界面

#### 功能目标
让玩家在选任务时就能感受到“自己正被哪些政治力量拉扯”。

#### 必显信息
- 任务名称
- 任务来源类型
- 发布者 / 请求方
- 关联人物或派系
- 预期收益
- 潜在政治意义

### 13.2 人物政治摘要区
建议在人物面板或主 HUD 摘要区常驻显示：

- 当前主要推荐人
- 当前主要阻力来源
- 当前候选资格提示
- 当前官职带来的权限摘要

### 13.3 势力政治总览页
建议聚焦展示：

- 主君与核心高层
- 主要派系块
- 玩家所处位置
- 当前势力重心
- 与玩家相关的重点职位或机会

### 13.4 月末政治报告界面

#### 必显信息
- 任务结果
- 数值变化
- 任命结果
- 原因分解
- 下月建议

#### 设计要求
- 一屏内读懂主要因果
- 优先显示结论，再显示原因
- 原因行不追求复杂公式，而追求人话化解释
- 不把玩家带入全局数据表

## 14. 数据与解释结构建议

### 14.1 核心新增定义方向
本阶段应正式冻结以下政治扩展结构：

- 推荐规则定义
- 反对规则定义
- 派系块定义
- 政治支持快照
- 候选评估结果
- 统一原因行结构

### 14.2 运行时最小状态要求
角色月末状态建议新增：

- 当前主要推荐列表
- 当前主要反对列表
- 当前派系态度摘要
- 当前可争职位候选资格
- 最近一次任命解释结果
- 下月政治机会标签

### 14.3 统一原因行结构要求
所有解释信息都建议统一为可复用的“原因行”对象，至少包含：

- 类型
- 来源人物 / 来源派系
- 影响方向
- 强弱等级
- 人话描述
- 所属结算阶段

这样可保证人物卡、势力页、月报共享同一套解释来源，而不是各自生成不同文案。

## 15. 与前后阶段的衔接

### 15.1 复用 Phase 2.1 的内容
Phase 3 必须直接复用以下基础能力：

1. 月初任务选择流程
2. 月内任务推进逻辑
3. 月末任务结算管线
4. 功绩与官职成长主轴
5. 基础月报与官职反馈 UI

### 15.2 对 Phase 4 的前置输出
Phase 3 完成后，应向后续阶段稳定输出：

1. 可复用的政治解释结构
2. 可复用的任命结果对象
3. 势力内部派系与支持链基础
4. 官职权限与信息可见性分层
5. 月末政治结算与下月建议机制

这将为家族、婚姻、门阀更深层介入政治博弈提供稳定接口。

---

## 16. 成功标准

本阶段完成后，以下内容必须为真：

1. 玩家每月能看到至少两类具有政治含义的任务来源。
2. 玩家能在人物摘要或月末报告中读到主要推荐人与主要阻力来源。
3. 任命结果至少能分解为资格、推荐、阻力、空缺、竞争中的若干原因。
4. 官职变化至少会影响任务可见性、信息可见性与一项政治后果反馈。
5. 势力页、人物页、月末报告三处界面能复用同一套政治解释数据。
6. 整个流程仍然保持单角色月度循环，不滑向传统君主全局 SLG。

---

## 17. 风险与控制

### 17.1 主要风险

#### 风险一：政治信息过多，玩家读不懂
控制方式：只展示与自己当前处境直接相关的支持、阻力、职位与建议，不做全量展开。

#### 风险二：政治结果看似可解释，实则仍然黑箱
控制方式：所有月末结论都必须能回溯到统一原因行，而不是只在 UI 拼描述。

#### 风险三：派系系统把玩法推向复杂 SLG
控制方式：派系先只做任命修正与关系倾向，不做资源运营与全局策略操作。

#### 风险四：官职变化仍然没有手感
控制方式：至少保证权限、信息可见性、待遇反馈、失败后果四项中有两项发生明显变化。

#### 风险五：月末报告过长，压垮节奏
控制方式：优先展示结果、主要原因与下月建议，详细原因可折叠或摘要化。

## 18. 开发优先级

### P0（必须完成）
- 推荐 / 反对 / 派系支持 / 任命解释正式合同
- 两类政治任务来源
- 月末政治结算原因分解
- 官职权限与信息可见性最小差异
- 月末政治报告与人物政治摘要

### P1（建议完成）
- 1–2 名 AI 候选竞争
- 势力政治总览页
- 政治失败类型细化
- 下月建议反馈

### P2（为下一阶段预留）
- 更深的门阀联动
- 婚姻带来的政治修正
- 多职位并发竞争
- 更复杂的派系议程系统

## 19. 测试要点

### 19.1 功能测试
- 月初候选任务是否稳定出现两类来源
- 推荐 / 反对是否会因关系与任务结果变化而变化
- 派系支持是否真实影响月末任命结果
- 任命失败是否能输出不同失败类别
- 官职变化是否改变权限与信息可见性

### 19.2 体验测试
- 玩家是否理解“谁在推我、谁在压我”
- 玩家是否理解“为什么成功 / 为什么失败”
- 玩家是否能回答“下月该做什么更有利”
- 玩家是否仍感到自己在扮演一个人，而非操纵整股势力

### 19.3 范围测试
- 势力页是否仍是摘要面板，而非全局控制台
- 高官职玩法是否仍遵守单角色行动约束
- 月末政治报告是否没有膨胀成复杂后台表格

---

## 20. 本章结论
Phase 3 的核心不是把项目变成更复杂的政治模拟器，而是把已建立的月度仕途骨架升级为“**可解释的仕途政治循环**”。

它要解决的关键问题是：让玩家不只知道自己有没有升官，而是知道自己为何得到机会、为何失去机会、该如何在单角色视角下继续经营政治命运。
