# 官职与任务原型部署数据字段设计 v1

#项目设计 #数据结构 #官职系统 #任务系统 #Phase2_1

> 文档定位：本文件用于为 Phase 2.1《官职与任务原型部署》冻结一版最小可用数据表结构，供 Godot Resource、运行时状态、任务结算与月末升官逻辑实现时参考。重点是字段语义清晰、依赖关系明确、方便后续 Phase 3 扩展。

---

## 1. 设计目标

### 1.1 本文档解决的问题
- Phase 2.1 需要新增哪些数据对象
- 每个对象至少包含哪些字段
- 哪些字段属于静态定义，哪些字段属于运行时状态
- 哪些字段必须为 Phase 3 预留扩展位

### 1.2 设计原则
1. 先保证“月初领任务—月内推进—月末升官”闭环。
2. 静态定义数据与运行时状态分离。
3. 用 ID 引用而不是深拷贝对象。
4. 字段设计优先服务原型可实现性，不预先复杂化。
5. 保留 Phase 3 所需的推荐、派系、政治解释扩展位。

---

## 2. 数据对象总览

Phase 2.1 建议新增或强化以下对象：

### 静态定义对象
- `OfficeData`：官职定义
- `TaskTemplateData`：任务模板定义
- `TaskPoolRuleData`：月初任务池生成规则
- `PromotionRuleData`：升官阈值与限制规则
- `CharacterSetupPatchData`：荀彧开局配置补丁

### 运行时状态对象
- `PlayerCareerState`
- `MonthlyTaskState`
- `MonthlyEvaluationResult`
- `TaskProgressSnapshot`

### 可复用旧对象但需补字段
- `CharacterData`
- `ScenarioData`

---

## 3. 静态定义与运行时状态分层

## 3.1 静态定义
静态定义建议存放在 `res://data/.../*.tres`，用于描述“有哪些官职、有哪些任务、哪些规则存在”。

特点：
- 可在 Inspector 中维护
- 可版本管理
- 不在运行时频繁改写

## 3.2 运行时状态
运行时状态建议存入存档 JSON，用于描述“玩家当前做到哪一步”。

特点：
- 每月变化
- 与角色进度强相关
- 用于界面显示与月末结算

## 4. OfficeData（官职定义）

## 4.1 作用
描述一个官职的层级、解锁内容、待遇文本与升迁位置，是本阶段仕途成长的核心定义对象。

## 4.2 最小字段
| 字段 | 类型 | 必须 | 说明 |
|------|------|------|------|
| id | string | 是 | 官职唯一 ID |
| name | string | 是 | 官职名称 |
| tier | int | 是 | 官职层级，数值越高代表层级越高 |
| category | string | 是 | 官职类别，如 civil / admin / central |
| description | string | 是 | 官职说明 |
| short_title | string | 否 | UI 简称 |
| unlock_task_tags | Array[string] | 是 | 解锁的任务标签 |
| blocked_task_tags | Array[string] | 否 | 禁止出现的任务标签 |
| merit_threshold | int | 是 | 升到本官职所需最低功绩 |
| min_months_in_prev_office | int | 否 | 前一官职最短任职月数 |
| stipend_text | string | 否 | 待遇或身份说明文本 |
| address_title | string | 否 | NPC 对玩家的称呼标签 |
| evaluation_bias | Dictionary | 否 | 月末评价偏向修正 |
| next_office_id | string | 否 | 下一阶官职 ID |
| prev_office_id | string | 否 | 前一阶官职 ID |
| is_phase2_1_available | bool | 是 | 是否在 Phase 2.1 可用 |
| sort_order | int | 是 | 列表排序值 |

## 4.3 建议的 evaluation_bias 结构
```text
evaluation_bias = {
  merit_gain_rate,
  fame_gain_rate,
  trust_gain_rate
}
```

## 4.4 原型样例
| id | name | tier | merit_threshold | next_office_id |
|----|------|------|------------------|----------------|
| office_commoner | 白身 | 0 | 0 | office_congshi |
| office_congshi | 从事 | 1 | 10 | office_zhubu |
| office_zhubu | 主簿级辅官 | 2 | 25 | office_central_aide |
| office_central_aide | 中枢幕僚级 | 3 | 45 |  |

## 5. TaskTemplateData（任务模板定义）

## 5.1 作用
描述一个可在月初被发放的任务模板，包括任务类型、完成条件、奖励结构和官职限制。

## 5.2 最小字段
| 字段 | 类型 | 必须 | 说明 |
|------|------|------|------|
| id | string | 是 | 任务模板 ID |
| name | string | 是 | 任务名称 |
| issuer_character_id | string | 是 | 发布人角色 ID |
| issuer_role_tag | string | 否 | 发布人身份标签，如 ruler / superior |
| task_type | string | 是 | 任务类型，如 logistics / politics / admin / personnel |
| task_tags | Array[string] | 是 | 任务标签，用于任务池筛选 |
| min_office_tier | int | 是 | 最低官职层级要求 |
| max_office_tier | int | 否 | 最高官职层级限制 |
| description | string | 是 | 任务说明 |
| objective_summary | string | 是 | 目标摘要 |
| duration_months | int | 是 | 默认持续月数，Phase 2.1 固定为 1 |
| difficulty | int | 是 | 难度等级 |
| recommended_stats | Dictionary | 否 | 推荐属性 |
| progress_rule_id | string | 是 | 任务进度判定规则 ID |
| success_condition | Dictionary | 是 | 成功条件定义 |
| excellent_condition | Dictionary | 否 | 优秀完成条件定义 |
| base_rewards | Dictionary | 是 | 标准奖励 |
| bonus_rewards | Dictionary | 否 | 优秀完成额外奖励 |
| fail_result | Dictionary | 是 | 失败结果 |
| followup_tags | Array[string] | 否 | 后续可解锁任务标签 |
| ui_priority | int | 否 | 月初任务卡片排序权重 |
| is_phase2_1_available | bool | 是 | 是否在 Phase 2.1 可用 |

## 5.3 建议奖励结构
```text

  merit,
  fame,
  trust
}

  merit,
  fame,
  trust
}

  merit_delta,
  fame_delta,
  trust_delta,
  failure_note
}
```

## 6. TaskPoolRuleData（月初任务池规则）

## 6.1 作用
控制每个月初向玩家展示哪些任务，用于把官职、角色状态、剧本阶段与任务模板关联起来。

## 6.2 最小字段
| 字段 | 类型 | 必须 | 说明 |
|------|------|------|------|
| id | string | 是 | 规则 ID |
| scenario_id | string | 否 | 适用剧本 ID，为空表示通用 |
| character_id | string | 否 | 指定角色 ID，如荀彧 |
| office_tier_min | int | 是 | 最低官职层级 |
| office_tier_max | int | 否 | 最高官职层级 |
| include_task_tags | Array[string] | 是 | 允许出现的任务标签 |
| exclude_task_tags | Array[string] | 否 | 排除的任务标签 |
| candidate_count | int | 是 | 月初提供候选任务数量 |
| weight_rules | Dictionary | 否 | 各类任务权重 |
| allow_repeat_in_consecutive_months | bool | 是 | 是否允许连续重复 |
| fallback_task_ids | Array[string] | 否 | 无候选时的保底任务 |
| is_phase2_1_available | bool | 是 | 是否用于 Phase 2.1 |

## 6.3 推荐规则
- Phase 2.1 默认候选数量：2–3 个
- 保底任务必须至少 1 个，避免月初无任务
- 连续重复任务建议默认关闭

---

## 7. PromotionRuleData（升官规则）

## 7.1 作用
用于控制月末升官判定，避免将阈值硬编码到多个系统中。

## 7.2 最小字段
| 字段 | 类型 | 必须 | 说明 |
|------|------|------|------|
| id | string | 是 | 规则 ID |
| from_office_id | string | 是 | 当前官职 ID |
| to_office_id | string | 是 | 目标官职 ID |
| required_merit | int | 是 | 所需功绩 |
| require_task_success_this_month | bool | 是 | 当月是否必须成功完成任务 |
| min_months_in_office | int | 否 | 当前官职最短任职月数 |
| min_trust | int | 否 | 最低信任要求，Phase 2.1 可不启用 |
| min_fame | int | 否 | 最低名望要求，Phase 2.1 可不启用 |
| blocker_tags | Array[string] | 否 | 阻塞标签 |
| success_note | string | 否 | 升官反馈文案模板 |
| failure_note | string | 否 | 未升官反馈文案模板 |
| is_phase2_1_available | bool | 是 | 是否用于 Phase 2.1 |

## 7.3 判定优先级建议
1. 当前是否已有下一官职
2. 当月任务是否成功
3. 功绩是否达标
4. 月数 / 信任 / 名望附加条件是否满足

## 8. CharacterSetupPatchData（荀彧开局配置补丁）

## 8.1 作用
用于在不重做整套剧本样本的前提下，快速将默认主角从曹操切换为荀彧，并覆盖其开局状态。

## 8.2 最小字段
| 字段 | 类型 | 必须 | 说明 |
|------|------|------|------|
| id | string | 是 | 补丁 ID |
| scenario_id | string | 是 | 适用剧本 ID |
| default_player_character_id | string | 是 | 默认玩家角色 ID，应为荀彧 |
| start_city_id | string | 是 | 初始所在城市 |
| start_faction_id | string | 是 | 初始势力 |
| start_office_id | string | 是 | 初始官职 |
| start_merit | int | 是 | 初始功绩 |
| start_fame | int | 是 | 初始名望 |
| start_trust | int | 是 | 初始信任 |
| start_relation_overrides | Array[Dictionary] | 否 | 与关键人物的初始关系覆盖 |
| start_flags | Array[string] | 否 | 开局标记 |

## 8.3 推荐初值
- `default_player_character_id = xun_yu`
- `start_office_id = office_congshi`
- `start_merit = 8~12` 或根据是否希望首月即可冲击升官决定
- `start_trust` 应高于普通幕僚，体现荀彧在曹操阵营中的定位

## 18. 实现建议

### 18.1 Godot 侧建议
- `OfficeData`、`TaskTemplateData`、`TaskPoolRuleData`、`PromotionRuleData` 使用自定义 `Resource`
- `PlayerCareerState`、`MonthlyTaskState`、`MonthlyEvaluationResult` 作为运行时 `Dictionary` 或专用状态类
- 存档仅保存运行时状态，不重复保存静态定义全文

### 18.2 UI 侧建议
- 月初任务界面直接读取 `TaskTemplateData + TaskPoolRuleData`
- 月末报告界面读取 `MonthlyEvaluationResult`
- 当前官职面板读取 `OfficeData + PlayerCareerState`

---

## 19. 本章结论
Phase 2.1 的数据设计重点不是“覆盖全部仕途政治复杂性”，而是用最少字段稳定支撑三个事实：

1. 玩家是谁，当前处于什么官职；
2. 本月接了什么任务，完成得如何；
3. 月末为何获得多少功绩，以及是否升官。

只要这三个事实被稳定建模，Phase 3 的推荐、派系、反对与可解释任命系统就可以在此基础上平滑扩展。
