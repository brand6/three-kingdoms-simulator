## 2. Phase 2.1 最小可玩闭环的录入目标

本阶段的数据录入不是为了覆盖整个 190 剧本，而是为了稳定跑通以下闭环：

```text
荀彧开局
→ 月初出现任务候选
→ 玩家选择 1 个主任务
→ 月内推进任务进度
→ 月末结算任务结果
→ 功绩 / 名望 / 信任变化
→ 满足阈值时触发升官
→ 下月任务池刷新
```

因此，所有数据录入都应服务于上面这条链，而不是追求内容全面。

## 5. 推荐录入顺序

## 5.1 先录规则骨架，再录世界样本

建议严格按以下顺序推进：

1. `OfficeData`
2. `PromotionRuleData`
3. `TaskTemplateData`
4. `TaskPoolRuleData`
5. `FactionData`
6. `CityData`
7. `CharacterData`
8. `CharacterSetupPatchData`
9. `ScenarioData`
10. `ClanData`（如启用安抚士族任务）
11. `RelationData`

### 原因
- 官职和升官规则是任务池与 CareerSystem 的前置；
- 任务模板需要先知道官职门槛和发布人；
- 荀彧开局补丁依赖角色、势力、城市、官职全部存在；
- 剧本入口应在全部核心样本都就绪后再收束。

## 6.4 CharacterData 录入要求

### 最小条数

#### D. 任务目标 / 配角组

## 6.8 TaskPoolRuleData 录入要求

### 最小条数

| `candidate_count` | 2 或 3 |
| `fallback_task_ids` | 至少 1 个保底任务 |

### 验收点

- 不会因过滤过严导致空池。

## 8. 录入完成后的验证顺序

### 验证 1：剧本入口
- 能否正确加载 190 原型剧本
- 默认玩家角色是否为荀彧

### 验证 2：官职与角色状态
- 荀彧初始官职是否为从事
- 当前功绩 / 名望 / 信任是否符合预期

### 验证 3：月初任务池
- 月初是否稳定出现 2–3 个候选任务
- 候选任务是否与当前官职匹配

### 验证 4：任务结算
- 至少 1 个任务能成功结算
- 至少 1 个任务能优秀完成或失败

### 验证 5：升官判定
- 功绩达标时能升官
- 官职变化后，下月任务池或文案发生变化

## 9. 数据录入时的常见错误

| 错误类型 | 表现 | 预防方式 |
|---|---|---|
| ID 不一致 | 任务发布人、官职、城市引用失败 | 统一 ID 命名并先录被依赖对象 |
| 任务池过滤过严 | 月初没有候选任务 | 每条规则必须配置 fallback_task_ids |
| 升官规则与官职阈值不一致 | UI 与实际结算矛盾 | 以 PromotionRule 为准，并校对 OfficeData 提示字段 |
| 角色初始值过高 | 首月无成长感 | 荀彧初始功绩建议略低于首轮升官线 |
| 样本录得太多 | 录入成本上升、验证变慢 | 先按“严格最小可跑条数”推进 |

## 11. 本章结论
《Phase 2.1 最小数据录入清单 v1》不是让开发 AI 一次性录完整个剧本，而是帮助它明确：

1. **哪些对象现在就必须录；**
2. **每类对象最少要录多少；**
3. **按什么顺序录最省返工；**
4. **录完如何验证闭环已经成立。**

只要先按本清单完成 P0 数据录入，Phase 2.1 就已经具备进入 Godot 原型开发与联调的最低条件。
