# 原型 UI 流程图 v1

#项目设计 #UI #UX #原型 #Godot

> 文档定位：本文件为原型阶段的 UI/UX 流程设计文档，重点不是美术风格，而是**信息架构、操作路径、界面职责与最小交互闭环**。供开发 AI 在 Godot 中搭建原型界面时使用。

---

## 1. 设计目标

原型 UI 需要解决四件事：

1. 让玩家快速理解“我是谁、我在哪、我能做什么”。
2. 让玩家在少量点击内完成一旬行动。
3. 让关系、仕途、士族、派系信息可被看见。
4. 让月末/事件反馈具有明确因果说明。

---

## 2. UI 总体结构

建议采用：

**主 HUD + 城市操作页 + 人物详情页 + 关系页 + 势力/派系页 + 事件弹窗**

### 2.1 一级入口
- 主 HUD
- 行动菜单
- 角色页
- 关系页
- 势力页
- 家族/士族页
- 事件/日志页

### 2.2 核心原则
- 关键行动不超过 3 次点击
- 关键状态常驻显示
- 面板职责单一，避免一页塞满全部信息

---

## 3. 主流程图（总览）

```text
启动游戏
  -> 剧本选择
  -> 人物选择/开局
  -> 主 HUD（所在城市）
      -> 打开行动菜单
          -> 选择行动
              -> 目标选择（如人物/城市）
                  -> 行动结果弹窗
                      -> 返回主 HUD
      -> 结束本旬
          -> 旬末结算弹窗
              -> 若到月末，进入月末评定/任命/事件
                  -> 返回主 HUD，进入下一旬
```

---

## 4. 主 HUD 设计

## 4.1 主 HUD 必须显示的信息

### 顶部信息条
- 当前时间：年 / 月 / 旬
- 当前城市
- 当前身份
- 所属势力
- 当前官职

### 左侧人物概览
- 姓名
- 头像占位
- AP
- 精力
- 压力
- 名望
- 功绩

### 右侧快捷信息
- 当前任务
- 最近事件提示
- 可进行的重要行动提示

### 底部主操作区
- 行动
- 角色
- 关系
- 势力
- 家族/士族
- 日志
- 结束本旬

## 4.2 主 HUD 的设计目标
- 玩家一进界面就知道现在状态
- 玩家不需要翻页就能决定本旬第一步做什么

---

## 5. 行动菜单流程

## 5.1 行动菜单结构

建议分 5 类：
- 成长
- 关系
- 政务
- 军事
- 家族

## 5.2 行动菜单流程图

```text
主 HUD
  -> 行动菜单
      -> 选择行动类别
          -> 查看可执行行动列表
              -> 查看行动说明/AP/精力消耗/条件
                  -> 选择目标
                      -> 确认执行
                          -> 结果弹窗
                              -> 返回主 HUD
```

## 5.3 行动列表项必须显示
- 行动名称
- 分类
- AP 消耗
- 精力消耗预估
- 目标类型
- 成功率/风险提示（如适用）
- 简短收益说明

---

## 6. 人物详情页

## 6.1 信息分区

### A. 基本信息
- 姓名 / 字 / 年龄 / 身份
- 所属势力 / 官职 / 所在城市

### B. 属性页签
- 六维属性
- 技能等级
- 特性标签

### C. 状态页签
- AP / 精力 / 压力 / 健康
- 名望 / 功绩 / 忠诚

### D. 社会关系页签
- 家族
- 士族
- 派系
- 亲族 / 配偶 / 子嗣 / 师友 / 仇敌

## 6.2 角色页作用
- 让玩家理解自己当前成长状态
- 作为“我要怎么规划路线”的决策页

---

## 7. 关系页

## 7.1 功能目标
关系页是本项目的重要核心页，需要支持：
- 查看关键人物关系
- 判断谁可结交、谁有风险、谁值得投资

## 7.2 关系列表推荐显示
- 人物名
- 当前关系标签
- 好感
- 信任
- 戒备
- 所属势力
- 可进行互动按钮

## 7.3 筛选维度
- 同势力
- 在城人物
- 亲族/姻亲
- 可拜访
- 可联姻
- 仇敌/高戒备

## 7.4 关系流转图

```text
主 HUD
  -> 关系页
      -> 选中目标人物
          -> 查看详细关系
              -> 执行互动（拜访/送礼/请教/议婚）
                  -> 结果弹窗
                      -> 关系列表刷新
```

---

## 8. 势力 / 派系页

## 8.1 势力页必须显示
- 势力名称
- 君主
- 城池数量
- 主要资源
- 当前主要方针（可简化）
- 主要干部

## 8.2 派系页必须显示
- 派系名称
- 核心人物
- 倾向（支持/反对/观望）
- 诉求标签
- 你与其关系状态

## 8.3 页面作用
- 告诉玩家“当前政治环境是什么”
- 避免任命失败、提案失败的黑箱感

---

## 9. 家族 / 士族页

## 9.1 页面目标
本页是项目特色页，必须让玩家看出“我不是孤立人物”。

## 9.2 家族页显示内容
- 家族名称
- 家主
- 家族声望
- 家族财富
- 家风标签
- 主要成员
- 当前家族需求/事件

## 9.3 士族页显示内容
- 士族名称
- 门第等级
- 区域影响力
- 倾向势力
- 友好/敌对士族
- 代表人物

## 9.4 典型流程

```text
主 HUD
  -> 家族/士族页
      -> 查看本家信息
          -> 查看可联姻对象 / 举荐线索 / 家族事件
              -> 发起互动
```

---

## 10. 事件弹窗

## 10.1 事件弹窗职责
- 呈现剧情结果
- 提供选择
- 告知数值变化与原因

## 10.2 事件弹窗必须显示
- 事件标题
- 事件正文
- 涉及人物头像/名字
- 选项列表
- 关键结果预告（如适合公开）

## 10.3 事件后反馈
事件处理后需明确显示：
- 关系变化
- 名望/功绩变化
- AP/精力/压力变化
- 新任务/新目标

---

## 11. 旬末 / 月末结算页

## 11.1 旬末结算页
显示：
- 本旬行动摘要
- 主要数值变化
- 关系变化摘要
- 新事件触发提示

## 11.2 月末结算页
显示：
- 功绩变化
- 任命/升迁/落选结果
- 势力资源变化摘要
- 家族/士族/派系影响摘要

## 11.3 设计重点
必须让玩家明白：
- 为什么升官/没升官
- 为什么某人对你好感变化
- 为什么压力升高

---

## 12. 原型核心交互路径

## 12.1 一旬标准路径

```text
进入主 HUD
  -> 查看 AP/状态/任务
  -> 行动 1
  -> 行动 2
  -> 打开关系页确认关键人物变化
  -> 行动 3/4
  -> 结束本旬
  -> 查看旬末结算
```

## 12.2 月末标准路径

```text
旬末结算
  -> 月末评定
      -> 任命/任务/事件
          -> 查看势力页或家族页
              -> 规划下月目标
```

## 14. 开发 AI 实现建议

### 14.1 先做功能通，再做美观
- 用 Panel/TabContainer/ScrollContainer 即可起步
- 先把信息与按钮跑通

### 14.2 UI 驱动逻辑，不反过来
- 所有按钮尽量调用统一系统接口
- 不在 UI 内写大量业务逻辑

### 14.3 可替换性
- 当前原型 UI 只是流程验证工具
- 后续可替换为更具沉浸感的布局与地图表现

---

## 15. 本阶段结论

《原型 UI 流程图 v1》定义了原型阶段最重要的界面原则：

1. 主 HUD 负责总览与决策入口
2. 行动菜单负责一旬主循环
3. 角色/关系/势力/家族页负责信息解释
4. 结算页与事件弹窗负责反馈与因果说明

只要这套信息流顺畅，原型即使没有复杂美术，也能准确验证玩法。
