# 发现与决策

## 需求
- 使用 Godot 引擎开发。
- 游戏基础体验参考《三国志10》。
- 融入《十字军之王》的一部分特性，而非纯三国志式体验。
- 项目核心应保持“单角色扮演、历史模拟、势力博弈”导向。
- 首个主剧本确定为 190 年剧本。
- 玩家可扮演全身份历史人物。
- 重点融合 CK 风格的士族门阀、联姻/子嗣、性格/关系、派系系统。
- 当前目标是先完成《项目总设计方案》。

## 研究发现
- 《三国志10》强项在于单武将视角、城市移动、任务驱动成长、舌战/单挑、仕官与自立。
- 《十字军之王》可借鉴的高价值特性通常包括：家族系统、婚姻/继承、性格与人际、封臣关系、派系与忠诚。
- 对 Godot 中小团队项目而言，最适合融合的是“关系与家族深度”，而非 CK 全量地图/封建契约仿真。
- 三国 190 剧本非常适合加入士族门阀：名门望族可影响仕途、婚配、举荐、忠诚、派系结构。
- “全身份”不应等于“全局策略游戏”，仍需坚持单角色约束，避免演变为纯君主 4X。
- 士族门阀系统可作为三国题材区别于普通武将 RPG 的核心特色层。
- 《核心系统详细设计》需优先定义时间结构、行动资源、人物数据、士族数据、关系网络、派系运转与身份边界，否则后续 GDD 会缺乏统一底层。
- 全身份设计的关键不是“给每个身份独立系统”，而是“同一底层规则在不同身份下开放不同权限与接口”。
- 士族门阀系统应优先提供：出身影响、举荐渠道、婚配池修正、派系吸附、名望传播，而不是一开始做超复杂宗族模拟。
- 派系系统首版应采用“半显性”设计：玩家能看到主要阵营、关系倾向、核心诉求，但不展示过多后台细值。
- 主循环文档需要回答三个关键问题：玩家每旬能做几件事、每项行动消耗什么、系统在何时统一结算。
- 数值骨架首版应优先定义“可感知资源”而非全部隐藏参数，避免设计失控。
- 三国单角色游戏的核心成瘾点来自“行动选择—关系反馈—职位变化—历史机会”链条，而非纯数值堆叠。
- 为兼顾可玩性与开发成本，首版数值系统应采用“少核心值、多修正项”的结构。
- 若已存在 AP，则精力不应再承担“限制行动次数”的职责，否则会形成重复约束。
- 精力最适合表现“短期状态与效率”，压力最适合表现“中长期心理负担”。
- 三国单角色体验中，宴饮、归家、探望亲族、休整等生活行动，需要依靠精力/压力系统才能获得稳定的玩法价值。
- 三资源模型中最重要的体验收益是：让人物不像永动机，并让战争、政治、社交与家庭生活产生连续状态联系。
- Godot 原型阶段最重要的是先验证“时间推进 + 行动选择 + 关系反馈 + 任命/成长”这条最小闭环，而不是一开始铺满全部历史内容。
- 原型开发应坚持“数据先行、界面从简、系统打通优先于表现精细”。
- 首版最容易失控的模块是全身份、战争表现、家族树深度和历史事件量，需明确分阶段接入。
- 对该项目而言，场景拆分应围绕“主界面 / 城市 / 人物 / 势力 / 战役 / 事件”六大交互入口，而不是按传统 RPG 地图堆场景。
- 面向开发 AI 的 GDD 不应只写概念描述，还要明确：模块边界、依赖关系、优先级、数据对象、验收标准与文档引用关系。
- GDD 框架的作用不是替代细化设计，而是作为开发索引与总装目录，帮助 AI 在后续实现时定位信息来源。
- Godot 原型实现前应先冻结一版最小数据模型，否则后续系统开发会频繁返工。
- 原型人物样本表不应追求史实全覆盖，而应优先覆盖“身份差异、士族差异、势力差异、派系差异”四类验证目标。
- 190 剧本原型最适合选取中原局部势力样本，因为该区域既有军政博弈，也有士族网络与名士活动空间。
- 原型 UI 的核心不是“美术布局”，而是“玩家是否能在 3 次点击内到达关键信息与关键行动”。
- 本项目 UI 需要突出人物、关系、势力、家族四类信息，不应让城市或地图信息掩盖单角色视角。
- 士族门阀系统若要成立，必须至少对以下玩法产生可感知影响：仕官、举荐、婚配、派系、名望传播、事件倾向。
- 士族系统不应只做“出身加成”，还需要支持“家族升降”与“新兴门第崛起”，否则长期游戏会失去动态性。
- 原型进入实现阶段后，最重要的是把“系统模块职责”和“任务执行顺序”写清，否则开发 AI 容易重复搭建或越层耦合。
- 事件样本表应优先服务于主循环验证，因此先做关系事件、任命事件、家族事件、势力任务事件，比先做大型历史战役更有效。
- 模块拆分文档应避免按纯代码文件维度写，而应按“职责 + 输入 + 输出 + 依赖 + 验收”维度写，便于 AI 实施。
- 当前主界面已具备顶栏状态、人物状态区、任务/事件区和底部导航，说明信息架构方向基本正确。
- 当前主界面最大问题不是“视觉不好看”，而是“核心主循环入口不够明确”，尤其缺少“结束本旬”与“推荐决策区”。
- 对本项目而言，主界面中心区域应该优先承载“本旬决策内容”，而不是样本说明文字。
- 家族/士族、派系、关键关系作为项目特色，至少应在主界面以摘要形式出现，否则设计辨识度不足。
- 行动菜单若直接平铺所有动作，随着身份、官职、地点、事件解锁会迅速失控，因此需要分层选择。
- 对该项目最合适的行动菜单结构是“固定一级分类 + 动态二级动作列表 + 按需目标选择”。
- 原型阶段行动分类应优先服务主循环，故建议先固定为：成长、关系、政务、军事、家族、移动/外出（可选）。
- 主界面改版时，最重要的不是“加更多信息”，而是“重新安排信息的优先级与位置”。
- 左栏应收缩为人物状态栏，中部应承担政治与社会摘要，右栏更适合承载任务/事件列表。
- 顶栏更适合显示时间、地点、身份、势力、士族等“抬头信息”，而官职可下沉到人物卡中。

## 技术决策
| 决策 | 理由 |
|------|------|
| 优先考虑“单角色 + 势力沙盘”架构 | 符合用户目标，也更适配《三国志10》基底 |
| CK 特性采用模块化裁剪方案 | 降低系统耦合，便于原型优先开发 |
| 将 CK 的“家族”本地化为“士族/门阀 + 家族”双层结构 | 更贴合东汉末年至三国政治生态 |
| 将派系系统嵌入势力内部权力斗争，而非独立大地图玩法 | 能保留 CK 味道，同时不破坏单角色体验 |
| 时间推进建议采用“旬/回合”结构 | 兼顾《三国志10》式行动感与历史模拟节奏 |
| 首版战争应采用“战略调度 + 局部战役”方案 | 降低 Godot 实现成本，突出角色作用 |
| 主循环以“旬”为主结算单位，“月”为中结算单位，“季/年”为大事件结算单位 | 有利于统一行动、成长、家族与历史事件节奏 |
| 三资源采用 AP（次数）+ 精力（短期效率）+ 压力（长期心境） | 分工明确，便于玩家理解与数值平衡 |
| Godot 原型采用“单场景主控 + 数据管理器 + UI 面板切换”架构起步 | 能快速验证系统闭环，避免过早陷入大地图与复杂场景组织 |
| GDD 框架需作为 AI 开发主入口文档 | 便于后续按模块持续补全设计、拆任务与执行实现 |
| 数据结构草案优先覆盖 Character/Faction/City/Clan/Family/Relation/Event | 这是原型阶段最核心的数据骨架 |
| 原型样本表优先覆盖 3~5 城、2~3 势力、30~50 人物 | 兼顾验证价值与数据录入成本 |
| 原型 UI 采用“主 HUD + 多信息面板”结构 | 最适合界面驱动原型与快速迭代 |
| 士族门阀系统 v2 采用“士族—家族—个人—关系网络”四层联动 | 便于把出身、婚配、举荐、派系与长期演化统一起来 |
| 原型实现阶段按“模块清单 -> 任务清单 -> 样本事件”顺序补文档 | 有利于开发 AI 直接进入实施 |
| 主界面评审优先级应按 P0/P1/P2 输出 | 便于开发 AI 先修正主循环阻塞问题，再优化识别度与质感 |
| 行动菜单的二级动作按身份、官职、地点、关系、状态动态过滤 | 能在保持结构稳定的同时支持全身份扩展 |
| 主界面改版应取消整体滚动，改为面板内滚动 | 可显著提升 HUD 稳定性与操作记忆 |

## Phase 2.1 关键发现
- 在 Phase 3 前单独插入 Phase 2.1 是合理的，因为“可解释政治”必须建立在稳定的任务、功绩、官职与月末结算输入之上。
- 将默认主角由曹操改为荀彧，能更好地验证文臣仕途成长、被委派任务、月末升官反馈与中枢政治入口，同时避免玩法滑向君主全局控制。
- Phase 2.1 的最小闭环应固定为：月初领任务 → 月内推进 → 月末结算 → 功绩变化 → 升官判定 → 下月任务池刷新。
- Phase 2.1 任务来源应先收敛为“上级指派任务”，这样最容易建立功绩归属、评价链路与 UI 节奏。
- Phase 2.1 升官判定应以功绩值为主，名望与信任只做辅助反馈，避免原型阶段过早进入复杂政治计算。
- 官职在 Phase 2.1 至少要改变三件事：可见任务池、NPC 称呼 / 身份反馈、月末评价文案；否则升官会缺乏手感。
- 最小数据录入顺序应固定为：OfficeData → PromotionRuleData → TaskTemplateData → TaskPoolRuleData → CharacterSetupPatchData → 必要角色 / 势力 / 城市补字段。
- Phase 2.1 运行时存档只需要保存动态状态：当前官职、累计功绩、当月任务、任务进度、月末评价结果；静态定义继续由 Resource 提供。
- 第一批最小样本足够支撑验证：4 级官职链、4 个任务模板、1 套荀彧前期任务池、1 套升官规则、1 个荀彧开局补丁。
- Phase 2.1 已新增两份核心文档：`design/总纲/官职与任务原型部署 Phase 2.1 v1.md` 与 `design/数据/官职与任务原型部署数据字段设计 v1.md`，并已同步 design 索引文件。
- 实现映射表的核心价值不是新增玩法，而是统一“系统职责、数据落点、UI 读取边界、月度时序和验收点”，它是正式进入开发前的桥接层。
- 对 Phase 2.1 而言，最关键的职责边界是：`TimeManager` 管时点、`TaskSystem` 管任务状态、`CharacterSystem` 管数值写入、`CareerSystem` 管升官判定、UI 只读结果不算逻辑。
- 若没有实现映射表，AI 最容易出现的错误是：把静态 Resource 和运行时状态混存、把升官判定写进 UI、或者让多个系统同时拥有月末结算权。
- “最小数据录入清单”不是重复字段设计，而是把字段稿、样本表和实现映射表压缩成可执行的数据录入顺序，避免一开始录太多、录太散、录到后面才发现缺关键依赖。
- 对 Phase 2.1 来说，真正的首批必录对象只有 9 类：Scenario、Faction、City、Character、Office、TaskTemplate、TaskPoolRule、PromotionRule、CharacterSetupPatch；Clan 和 Relation 是为任务可玩性补足的第二批。
- 数据录入应先录规则骨架（Office / Promotion / Task / TaskPool），再录世界样本（Faction / City / Character / Patch / Scenario），最后补 Clan / Relation；这样返工最少。

## Phase 3 规划关键发现
- Phase 3 不能绕开 Phase 2.1 另起一套政治循环，而应直接复用“月初任务池—月内推进—月末结算—官职变化”的月度骨架。
- “可解释政治”成立的前提不是增加更多隐藏权重，而是让推荐、反对、派系支持、资格不足和机会错失都能回溯到可见来源。
- Phase 3 的第一步必须先冻结政治扩展数据契约，否则后续推荐链、派系支持和任命解释会继续依赖临时 Dictionary 字段。
- Phase 3 至少要把任务来源从“上级指派”扩到两个原型来源，推荐组合应优先采用“势力指派 + 人际/士族请求”这类可解释来源，而不是一次性铺满全部事件池。
- 推荐 / 反对链应复用已有关系、信任、功绩、官职与任务结果，不应重建并行政治数值体系。
- 派系支持演算应先做“可见倾向 + 月末修正 + 原因明细”，不宜直接进入 CK 式复杂派系契约。
- 官职在 Phase 3 中不只影响任务池，还必须开始影响权限、发言权、候选资格和失败时的政治后果文案。
- Phase 3 的最小 UI 范围应聚焦于三处：月末政治报告、人物政治支持摘要、势力/派系总览；不应先做大而全的全局政治地图。
- 将 Phase 3 拆为 7 个 plans 后，执行顺序可稳定收敛为：契约层 → 来源层 → 关系链层 → 任命判定层 → 权限后果层 → UI 层 → 联调验收层。
- 本次规划无需新增 `design/` 文档索引，因为没有创建新的 `design/` 文档，新增内容全部位于 `.planning/`。

## Phase 3 文档落盘关键发现
- Phase 3 正式总纲文档应继续沿用 Phase 2.1 的章节式 GDD 写法，而不是改写成实现清单，这样最适合作为后续 plans、实现映射与 UI 验收的稳定上位文档。
- Phase 3 的设计重心必须聚焦“单角色如何理解并经营政治处境”，而不是“势力如何被玩家全局管理”。
- 势力信息可见性在本阶段应坚持“摘要化、因果化、与玩家相关”，不直接暴露全势力后台数值表。
- 推荐 / 反对 / 派系支持 / 任命解释四类信息必须共享统一原因行结构，否则人物页、势力页与月报会再次各说各话。
- 官职在 Phase 3 中至少要同时影响任务来源、信息可见性、发言资格与失败后果，才能让升迁真正有手感。
- 月末政治结算必须先给结论，再给原因，再给下月建议，避免报告可读性被复杂政治说明拖垮。
- `design/总纲/Phase 3 仕途、势力与可解释政治 详细规划 v1.md` 已落盘，并已同步更新 `design/Agent.md`、`design/machine_index.json`、`design/CHANGELOG.md` 与 `.planning/ROADMAP.md`。
- 新增：`design/数据/Phase 3 ID 与样本命名冻结表 v1.md`，用于冻结首批样本 ID 命名、交叉引用與支撑占位 ID，解决命名漂移與样本未冻结的风险，便于数据录入（已同步 design 索引）。

## Phase 3 字段稿与实现映射表落盘关键发现
- Phase 3 的第一优先级不是先写 resolver 逻辑，而是先冻结推荐规则、反对规则、派系块、政治支持快照、候选评估与原因行的数据合同，否则实现层会继续散落在临时字段里。
- `MonthlyEvaluationResult` 应继续作为月报唯一入口，只扩政治字段，不另起平行报告对象；这样 Phase 2.1 到 Phase 3 的月度闭环能平滑延续。
- `PoliticalReasonLine` 必须成为人物页、势力页与月报三处 UI 的共用解释单元，否则“可解释政治”会再次退化为各面板各写一套文案。
- Phase 3 的实现映射重点不是增加模块数量，而是把 `PoliticalSystem`、`AppointmentResolver`、`CareerSystem`、`FactionSystem` 与 UI 的读取边界切清，避免把任命判定写进面板或把政治支持分散到多个系统。
- `design/数据/Phase 3 政治与任命数据字段设计 v1.md` 与 `design/原型与实现/Phase 3 Godot 实现映射表 v1.md` 已落盘，并已同步更新 design 索引与根规划记录。

## Phase 3 最小数据录入清单关键发现
- Phase 3 的首批录入重点不再是继续补更多人物，而是先保证 `RecommendationRuleData`、`OppositionRuleData`、`FactionBlocData` 与两类任务来源样本都已齐备，否则 explainable-politics 只会停留在字段层。
- `PoliticalSupportSnapshot` 虽然是运行时对象，但必须反推其世界依赖样本来做录入清单；如果没有推荐人、反对者、派系块、士族与竞争关系，月报就不会有真正可解释的内容。
- `AppointmentCandidateEvaluation` 至少需要一组玩家胜出样本和一组玩家落选样本，否则无法验证任命解释是否真的覆盖资格、推荐、阻力、空缺与竞争几类原因。
- `MonthlyEvaluationResult` 在 Phase 3 中必须至少准备 1 条成功月报和 1 条失败月报样本，且两者原因结构明显不同，才能验证“成功 / 失败不是同一套模板换字”。
- Phase 3 的数据录入清单必须继续坚持单角色范围，只围绕荀彧在曹操势力内部的最小政治样本圈展开，不扩成全国或全势力全官员录表。

## Phase 3 首批政治样本名单关键发现
- Phase 3 仍然缺少一份“字段之外、可直接录入”的首批政治样本名单；若只靠字段稿与录入清单，实际数据录入时仍会反复犹豫该先录哪些具体条目。
- 首批政治样本必须继续收缩到“荀彧—曹操势力内部政治圈”，并且外部只保留 1 个最小对照势力；否则样本会从 explainable-politics 原型滑回剧本扩量。
- 推荐规则、反对规则、派系块、关键人物、竞争样本、任务来源样本、成功 / 失败月报样本这 7 组是 Phase 3 首批样本的最小稳定分组，缺任何一组都会让月报解释链断裂。
- 最合适的首批人物网不是继续补全曹操全员，而是固定主君、同门推荐人、名士观察位、主战保留位、旧吏留任位、竞争者与最小对照势力位。
- 成功月报与失败月报必须作为首批静态样本名单的一部分提前冻结，否则联调阶段很容易只验证成功路径，遗漏落选与保留的解释质量。
