# 发现与决策

## 需求
- 使用 Godot 引擎开发。
- 游戏基础体验参考《三国志10》。
- 融入《十字军之王》的一部分特性，而非纯三国志式体验。
- 项目核心应保持“单角色扮演、历史模拟、势力博弈”导向。
- 首个主剧本确定为 190 年剧本。
- 玩家可扮演全身份历史人物。
- 重点融合 CK 风格的士族门阀、联姻/子嗣、性格/关系、派系系统。
- 当前目标是先完成《项目总设计方案》。

## 研究发现
- 《三国志10》强项在于单武将视角、城市移动、任务驱动成长、舌战/单挑、仕官与自立。
- 《十字军之王》可借鉴的高价值特性通常包括：家族系统、婚姻/继承、性格与人际、封臣关系、派系与忠诚。
- 对 Godot 中小团队项目而言，最适合融合的是“关系与家族深度”，而非 CK 全量地图/封建契约仿真。
- 三国 190 剧本非常适合加入士族门阀：名门望族可影响仕途、婚配、举荐、忠诚、派系结构。
- “全身份”不应等于“全局策略游戏”，仍需坚持单角色约束，避免演变为纯君主 4X。
- 士族门阀系统可作为三国题材区别于普通武将 RPG 的核心特色层。
- 《核心系统详细设计》需优先定义时间结构、行动资源、人物数据、士族数据、关系网络、派系运转与身份边界，否则后续 GDD 会缺乏统一底层。
- 全身份设计的关键不是“给每个身份独立系统”，而是“同一底层规则在不同身份下开放不同权限与接口”。
- 士族门阀系统应优先提供：出身影响、举荐渠道、婚配池修正、派系吸附、名望传播，而不是一开始做超复杂宗族模拟。
- 派系系统首版应采用“半显性”设计：玩家能看到主要阵营、关系倾向、核心诉求，但不展示过多后台细值。
- 主循环文档需要回答三个关键问题：玩家每旬能做几件事、每项行动消耗什么、系统在何时统一结算。
- 数值骨架首版应优先定义“可感知资源”而非全部隐藏参数，避免设计失控。
- 三国单角色游戏的核心成瘾点来自“行动选择—关系反馈—职位变化—历史机会”链条，而非纯数值堆叠。
- 为兼顾可玩性与开发成本，首版数值系统应采用“少核心值、多修正项”的结构。
- 若已存在 AP，则精力不应再承担“限制行动次数”的职责，否则会形成重复约束。
- 精力最适合表现“短期状态与效率”，压力最适合表现“中长期心理负担”。
- 三国单角色体验中，宴饮、归家、探望亲族、休整等生活行动，需要依靠精力/压力系统才能获得稳定的玩法价值。
- 三资源模型中最重要的体验收益是：让人物不像永动机，并让战争、政治、社交与家庭生活产生连续状态联系。
- Godot 原型阶段最重要的是先验证“时间推进 + 行动选择 + 关系反馈 + 任命/成长”这条最小闭环，而不是一开始铺满全部历史内容。
- 原型开发应坚持“数据先行、界面从简、系统打通优先于表现精细”。
- 首版最容易失控的模块是全身份、战争表现、家族树深度和历史事件量，需明确分阶段接入。
- 对该项目而言，场景拆分应围绕“主界面 / 城市 / 人物 / 势力 / 战役 / 事件”六大交互入口，而不是按传统 RPG 地图堆场景。
- 面向开发 AI 的 GDD 不应只写概念描述，还要明确：模块边界、依赖关系、优先级、数据对象、验收标准与文档引用关系。
- GDD 框架的作用不是替代细化设计，而是作为开发索引与总装目录，帮助 AI 在后续实现时定位信息来源。
- Godot 原型实现前应先冻结一版最小数据模型，否则后续系统开发会频繁返工。
- 原型人物样本表不应追求史实全覆盖，而应优先覆盖“身份差异、士族差异、势力差异、派系差异”四类验证目标。
- 190 剧本原型最适合选取中原局部势力样本，因为该区域既有军政博弈，也有士族网络与名士活动空间。
- 原型 UI 的核心不是“美术布局”，而是“玩家是否能在 3 次点击内到达关键信息与关键行动”。
- 本项目 UI 需要突出人物、关系、势力、家族四类信息，不应让城市或地图信息掩盖单角色视角。
- 士族门阀系统若要成立，必须至少对以下玩法产生可感知影响：仕官、举荐、婚配、派系、名望传播、事件倾向。
- 士族系统不应只做“出身加成”，还需要支持“家族升降”与“新兴门第崛起”，否则长期游戏会失去动态性。
- 原型进入实现阶段后，最重要的是把“系统模块职责”和“任务执行顺序”写清，否则开发 AI 容易重复搭建或越层耦合。
- 事件样本表应优先服务于主循环验证，因此先做关系事件、任命事件、家族事件、势力任务事件，比先做大型历史战役更有效。
- 模块拆分文档应避免按纯代码文件维度写，而应按“职责 + 输入 + 输出 + 依赖 + 验收”维度写，便于 AI 实施。

## 技术决策
| 决策 | 理由 |
|------|------|
| 优先考虑“单角色 + 势力沙盘”架构 | 符合用户目标，也更适配《三国志10》基底 |
| CK 特性采用模块化裁剪方案 | 降低系统耦合，便于原型优先开发 |
| 将 CK 的“家族”本地化为“士族/门阀 + 家族”双层结构 | 更贴合东汉末年至三国政治生态 |
| 将派系系统嵌入势力内部权力斗争，而非独立大地图玩法 | 能保留 CK 味道，同时不破坏单角色体验 |
| 时间推进建议采用“旬/回合”结构 | 兼顾《三国志10》式行动感与历史模拟节奏 |
| 首版战争应采用“战略调度 + 局部战役”方案 | 降低 Godot 实现成本，突出角色作用 |
| 主循环以“旬”为主结算单位，“月”为中结算单位，“季/年”为大事件结算单位 | 有利于统一行动、成长、家族与历史事件节奏 |
| 三资源采用 AP（次数）+ 精力（短期效率）+ 压力（长期心境） | 分工明确，便于玩家理解与数值平衡 |
| Godot 原型采用“单场景主控 + 数据管理器 + UI 面板切换”架构起步 | 能快速验证系统闭环，避免过早陷入大地图与复杂场景组织 |
| GDD 框架需作为 AI 开发主入口文档 | 便于后续按模块持续补全设计、拆任务与执行实现 |
| 数据结构草案优先覆盖 Character/Faction/City/Clan/Family/Relation/Event | 这是原型阶段最核心的数据骨架 |
| 原型样本表优先覆盖 3~5 城、2~3 势力、30~50 人物 | 兼顾验证价值与数据录入成本 |
| 原型 UI 采用“主 HUD + 多信息面板”结构 | 最适合界面驱动原型与快速迭代 |
| 士族门阀系统 v2 采用“士族—家族—个人—关系网络”四层联动 | 便于把出身、婚配、举荐、派系与长期演化统一起来 |
| 原型实现阶段按“模块清单 -> 任务清单 -> 样本事件”顺序补文档 | 有利于开发 AI 直接进入实施 |
