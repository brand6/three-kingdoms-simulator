## 关键问题
1. 190 剧本中，是否只覆盖中原主要势力，还是首版就覆盖全国全城市与全部历史人物？
2. 全身份里，首版是否允许玩家从“君主身份”直接完整操控全部内政/外交，还是保留单角色约束下的有限操作？
3. 士族门阀系统更强调“出身资源差异”，还是“长期政治影响与婚姻网络”？
4. 项目总设计方案完成后，下一步是继续细化 GDD，还是直接拆原型模块？

## 已做决策
| 决策 | 理由 |
|------|------|
| 先进行高层设计与需求澄清，再展开完整 GDD | 用户需求方向明确，但历史范围与特色系统取舍尚未确定 |
| CK 特性需做轻量化融合，而非全量照搬 | Godot 单机项目需控制复杂度，确保可落地 |
| 首个主剧本定为 190 年群雄割据 | 该时期最适合单角色崛起与多势力博弈 |
| 玩家身份范围定为全身份 | 用户明确希望可扮演不同历史人物 |
| CK 融合重点为士族门阀、联姻子嗣、性格关系、派系 | 与三国时代特征契合，且具差异化价值 |
| 当前产出物先对齐为《项目总设计方案》 | 用户当前目标是需求对齐与高层设计 |
| 下一份核心文档为《核心系统详细设计 v1》 | 便于把高层框架落到可开发的模块定义 |
| 当前继续输出《主循环与数值骨架 v1》 | 需要明确游戏节奏、资源流与各系统结算逻辑 |
| 保留 AP + 精力 + 压力三资源结构 | 能同时表达行动次数、短期状态与中长期心理负担 |
| 精力定位为“效率资源”而非“行动门票” | 避免与 AP 重叠，减少双重限制带来的挫败感 |
| 当前进入《Godot 原型开发拆解 v1》 | 需要把设计文档转换成可执行的开发阶段与模块计划 |
| 《GDD 框架 v1》面向开发 AI 输出 | 需要把设计内容组织成可拆分、可引用、可执行的标准文档结构 |
| 《Godot 数据结构草案 v1》作为实现前置文档 | 便于开发 AI 先稳定数据对象，再搭建系统与 UI |
| 《190 剧本原型人物/势力样本表 v1》采用局部样本策略 | 先验证系统闭环，不追求全国全人物覆盖 |
| 《原型 UI 流程图 v1》以界面驱动原型为前提 | 原型阶段优先验证信息流与操作流，而非沉浸式场景表现 |
| 士族门阀系统进入 v2 专项细化 | 该系统是项目核心特色，需要从“有概念”提升到“可落地规则” |
| 《Godot 系统模块拆分清单 v1》按管理层/系统层/UI层拆分 | 便于开发 AI 并行实现与控制依赖 |
| 《原型任务拆解清单 v1》采用任务包与验收标准结构 | 便于直接转为开发执行单 |
| 《190 剧本原型事件样本表 v1》优先覆盖关系/任命/家族/势力四类事件 | 先验证循环反馈，再扩历史主线 |
| 主界面现状属于“原型骨架可用，但未达可玩主 HUD 标准” | 已具备状态展示，但缺少结束本旬、决策核心区与政治摘要 |
| 行动菜单采用“先大类，再子类”两级结构 | 最适合多身份、强状态、多行动来源的单角色历史模拟原型 |
| 主界面改版方向采用“窄左栏 + 中部三摘要 + 右侧双列表 + 底部主操作” | 更符合单角色主 HUD 的扫读效率与项目特色表达 |
| 在 Phase 3 前插入 Phase 2.1 | 先完成任务—功绩—官职闭环，再接入可解释政治更稳妥 |
| Phase 2.1 默认主角改为荀彧 | 比曹操更适合验证单角色仕途成长、任务委派与升官反馈 |
| Phase 2.1 先只做上级指派任务 | 逻辑清楚、功绩归属明确、便于月末评价与升官原型验证 |
| Phase 2.1 升官判定以功绩为主，名望与信任先做辅助反馈 | 控制复杂度，避免过早引入复杂任命政治演算 |
| Phase 2.1 文档分为总纲章节 + 数据字段稿两份 | 便于分别服务设计对齐与实现建模 |
